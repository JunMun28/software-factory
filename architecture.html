Identity & enforcement

role Submitter

Microsoft SSO only. Files Requests, watches their own. Never touches GitHub.

role Admin

SSO + links GitHub once. Approves the gates as themselves from the control center — real human in GitHub's history; can't approve own work (GitHub-enforced).

identity Factory Builder bot

One GitHub App. Only writes — repos, PRs, commits, progress comments. Never approves (an App can't approve its own PR). Not the Actions token.

Where rules are guaranteed

Enforcement lives in GitHub (CI checks + protected branches), not in agent permissions and not in the web app. The web app is a friendly front-end on top of GitHub's gates. The coder agent physically cannot weaken its tests because CI rejects any implementer diff that touches tests/ — the load-bearing test-isolation gate.

Stack & data flow

Stack

  • Backend — FastAPI + uv (Python)
  • Frontend — Angular (SPA)
  • DB — SQLite (local) → Azure SQL / MSSQL, via SQLAlchemy + Alembic
  • LLM — OpenAI raw API → corporate wrapped LLM, behind one LLMClient
  • Host — Azure App Service + Azure SQL; agents in GitHub Actions
  • GitHub — one Factory App (installation token = bot; per-admin user token = approve)

Two "swap-later" seams

Both isolated behind an interface so the corporate/managed version drops in with one change:

  • SQLite → MSSQL — SQLAlchemy layer; stay DB-agnostic, no SQLite-isms.
  • OpenAI → corporate LLM — LLMClient adapter; likely just a base-URL change.

Live updates: GitHub webhook → FastAPI → DB → Angular polls (~3–5s now; SSE later).

Decisions on record

ADR 0001
Copilot-first; test-isolation enforced at the CI/git layer

Copilot for reach; the guarantee bought back at CI since Copilot isn't a sandbox.

ADR 0002
PR-centric, event-driven orchestration; gates = native GitHub controls

Work item = branch + PR. Build the inner loop (3→4→5) first.

ADR 0003
Draft spec lives in the DB until approval

New apps have no repo yet; approval is the bridge that creates the repo + SPEC.md.

ADR 0004
Progress reporting via milestone summaries over GitHub events

PR-comment summaries on the same rail as gates — not a live agent-callback stream.

ADR 0005
Control center acts through GitHub; deploy via a protected production branch

Verified: environment reviewers/custom rules aren't available for private repos on Team. Admins approve as themselves.

ADR 0006
Every Stage is resumable

Re-enter from any stage on current PR state — Retry / Send back / Take over depend on it.

ADR 0007
Web app stack: FastAPI + Angular, SQLite→MSSQL, OpenAI→corporate, Azure, one GitHub App

Two swap-later seams; pre-approval AI runs in the web app, Stages 2–6 on Copilot in CI.
