# MSSQL timezone CI fix report

## Status

PASS — the ORM now exposes every persisted model timestamp as timezone-aware UTC on SQLite and MSSQL while storing naive UTC in the databases' existing `DATETIME` representation.

## Root cause and fix

- `DateTime(timezone=True)` did not restore timezone information from MSSQL/pyodbc's plain `DATETIME` values.
- Added `TZDateTime`, a `TypeDecorator` over `DateTime` that rejects naive input, converts aware input to naive UTC on bind, and returns aware UTC on load.
- Replaced all 11 timestamp column declarations in `api/app/models.py` with `TZDateTime`.
- Updated the scoped-recovery clock assertion to compare two aware timestamps.
- Added one SQLite round-trip regression test; SQLite reproduces the offset-naive database result and therefore pins the MSSQL ORM guarantee.

## Timestamp hazard survey

- `api/app/supervision.py`: loaded `ProgressEvent.created_at` and `Request.stage_entered_at` values now arrive aware UTC; its compatibility normalization remains safe and arithmetic with `utcnow()` is uniform.
- `api/app/routers/mission.py`: the aware `week_ago` query parameter is normalized to naive UTC by `TZDateTime` before binding.
- `api/app/intents.py`, `api/app/transitions.py`, and seed paths already create aware values and now pass through the same bind normalization.
- No remaining `DateTime(timezone=True)` declarations exist in `api/app/models.py`.
