# Making the Fabric intake interview responsive

> Research report — 2026-07-07. Produced by a 16-agent workflow (3 map → 6 research → 6 adversarial-verify → 1 synthesize). Every external claim was checked against Anthropic docs; every code claim carries a file:line.

> **Decision (2026-07-07): the Haiku model-swap (old Phase 2) is REJECTED — question quality is the priority, and Haiku risks weaker follow-ups on deep interviews.** Keep Sonnet 5 for `next_question`. Responsiveness therefore comes entirely from Phase 1 (pre-generation + streaming), which hides latency without touching the model or its output quality. The one quality-preserving speed lever — disabling Sonnet's extended thinking for the narrow question step — is coupled to the deferred direct-API path, so it's a later experiment, not part of this plan.

> **UPDATE (2026-07-07) — the biggest lever was hidden, and it's neither streaming nor a model swap.** Probing the actual `claude -p` call (Claude Code 2.1.187) with `--include-partial-messages` revealed that each intake call loads the **host's full Claude Code environment** — the `SessionStart` hook fires and Sonnet burns *extended-thinking tokens deliberating "should I invoke a skill?"* before it ever addresses the request. That overhead is **~40% of the wall-clock** and is pure noise for a scoped intake question. Fix (shipped): pass **`--safe-mode`** on the read-only intake calls (drops host skills/hooks/CLAUDE.md/MCP; keeps model, built-in tools, and auth). Measured on Sonnet 5, same prompt, quality preserved: **baseline 25.9s → `--safe-mode` 15.9s (−39%) → `--safe-mode --effort low` 10.2s (−61%)**. `--effort` is wired as an off-by-default knob (`FACTORY_INTERVIEW_EFFORT`) since it's a quality dial. Net: `--safe-mode` is a quality-**free** win that beats streaming for responsiveness at a fraction of the effort — and it makes the invisible-thinking problem that limited streaming much smaller. **Implemented:** [interview_gen.py](../../api/app/interview_gen.py) (Phase-1 non-blocking + background pre-gen), and `--safe-mode` in [agent_exec.py](../../api/app/agent_exec.py). Streaming is now optional polish over a ~15s wait rather than the main event.

## 1. TL;DR

Ship a **two-phase change, both grounded in one net-new SSE endpoint**. Phase 1 (perceived latency, ~2-3 days): pre-generate the next interview question in an in-process background thread the moment the user answers, split `answer_interview` so the composer never blocks, and stream the question to the UI token-by-token as it generates. This turns the current "stare at three dots for 60-120s" into a chat that types back within seconds, with **zero change to the model round-trip**. Phase 2 (actual latency, ~1-2 days, do it in the same PR or immediately after): swap the per-question model from launch.json's Sonnet 5 to **Haiku 4.5** for `next_question` only, keeping Sonnet 5 for `draft_spec`. That cuts real generation time ~30-50% and buys huge timeout headroom, essentially free. Everything stays behind the ADR 0007 brain seam, preserves the codex + scripted paths, and respects single-worker uvicorn.

## 2. Where the time actually goes today

The interview is a **fully synchronous request/response cycle**. Every question is generated by a blocking CLI subprocess call made *inside* the HTTP handler, and the Angular UI awaits one plain HTTP response — it does not poll or stream.

**Actual latency — the wall-clock cost, per question:**

- **Model round-trip is the dominant sink.** The code's own comment budgets 60-90s and sets a 120s timeout because "a cold `claude` CLI on a larger model routinely runs 60-90s" (`api/app/settings.py:23-26`). This is one prompt-in / one-answer-out and it is *unchanged* by any caching, pooling, or transport trick — only a faster model or a smaller prompt moves it.
- **CLI cold-start stacks on top.** Each question is one whole new cold OS process — `subprocess.Popen(cmd, ..., start_new_session=True)` (`api/app/agent_exec.py:90`), blocked on `proc.communicate(timeout=...)` (`api/app/agent_exec.py:95`). No daemon, no `--resume`, no session, no pool anywhere in `api/app`, so node/rust boot + auth/config read + MCP/tool init is paid fresh every question (`api/app/agent_brain.py:103` → `run_agent`). Empirically ~seconds (the repo's own smoke test in `docs/superpowers/specs/2026-07-05-real-claude-cli-intake-interview-design.md` shows ~8.1s TTFT on a clean cwd).
- **Cost repeats per question, not per interview.** `next_question` runs once per follow-up; budget ceilings are 3 / 5 / 10 (`api/app/interview.py:17-22`). A "new app" interview pays cold-start + round-trip up to **10 times, fully serial** (`api/app/agent_brain.py:70-116`).
- **Throwaway workdir setup on the critical path every call.** `_run_with_attachments` does `mkdtemp` + (when attachments exist) `shutil.copyfile` each attachment + `rmtree` per question (`api/app/agent_brain.py:48-64`, `api/app/attachments.py:104,117-118`). Deliberate (keeps the CLI from discovering the repo's CLAUDE.md/skills), but re-run every time.
- **JSON parsing is negligible** (`api/app/agent_exec.py:131`, `api/app/agent_brain.py:104`) — microseconds vs 60-90s.

**Perceived latency — what the user actually experiences:**

- **The HTTP handler blocks on the model, twice per pair.** `GET /interview` calls `current_question` → `get_brain().next_question(r)` inline (`api/app/routers/requests.py:42-53,157-159`). `POST /interview` (answering) records the turn, then returns `interview_state(...)` which re-runs `current_question` → generates the *next* question and blocks again before the POST returns (`api/app/routers/requests.py:162-174`). Generate-once persistence (`requests.py:47-52`) prevents *duplicate* generation, not the blocking wait.
- **The UI is one-shot, no streaming, no polling.** The constructor subscribes once to `api.interview(id)` and clears `busy` on the single response (`apps/intake/src/app/submitter/interview.ts:673-685`); `push()` POSTs once via `api.answer` (`interview.ts:715-731`); `api.service.ts:63-68` are plain one-shot `http.get`/`http.post`. The only feedback is an indeterminate "reading your answer" three-dot typing row (`interview.ts:76-83`). The user stares at dots for the entire round-trip.
- **There is NO streaming infra to reuse.** `ws_exec.py` is a misnomer — 32 lines of `_git`/`_pytest` `subprocess.run` wrappers, zero WebSocket code. A grep for `WebSocket|StreamingResponse|text/event-stream` across `api/app` returns nothing. This plan **builds the first one.**

**The one-line summary:** the dominant *actual* cost is the model round-trip (60-90s), which only a faster model touches; the dominant *perceived* cost is that the user sees nothing until that entire round-trip finishes, twice, which only streaming + pre-generation touch.

## 3. Ranked options

| Approach | Actual-latency impact | Perceived impact | Effort | Risk | Verdict |
|---|---|---|---|---|---|
| **predictive-ui** (pre-generate next Q in background thread + optimistic UI + non-blocking composer) | None directly (hides one of two round-trips behind think-time; pre-warms Q1 at request create) | **Large** — composer never blocks; answer appears instantly, next Q lands mostly during think-time | Moderate | Low-medium (concurrency races, optimistic rollback, timeout tail) | **holds** |
| **stream-tokens** (SSE endpoint, `claude --output-format stream-json --include-partial-messages`) | ~Zero (same process, streamed read) | **Large** — question renders word-by-word instead of after a blank wait | Medium (4 layers: subprocess, brain, async SSE endpoint, Angular client) | Medium (contract change to human-readable-first, codex can't stream, single-worker event-loop, SSE buffering) | **holds** |
| **model-latency** (Haiku 4.5 for `next_question`, Sonnet 5 for `draft_spec`) | **Real, ~30-50% of the model term**; big timeout headroom; ~3x cheaper | Modest alone (shorter wait, still behind dots) | Very low (env knob + thread `model=` param) | Low (answer-quality drift on deep "new app" interviews; import-time binding foot-gun) | **holds technically — REJECTED by decision: question quality over speed** |
| **direct-api-not-cli** (Anthropic Messages API behind the seam, replaces `claude -p`) | Removes cold-start + workdir (seconds); enables clean streaming + caching + `effort:low`/`thinking:disabled` | Large *only with* the streaming half | Small-moderate for transport; **large** for the streaming/UI half | Medium (auth in container, answer-quality drift vs CLI system prompt, per-token billing) | **partial** — sound, but headline oversells; win is smaller-and-differently-sourced than "removes 60-120s" |
| **warm-worker** (persistent `ClaudeSDKClient` / pool) | Removes cold-start only (not the round-trip) | ~None alone (still need streaming) | High (async-only SDK vs sync routers, lifecycle, pool, breaks codex seam) | High (SDK "not built for multi-tenant servers", single-shared-client can't isolate concurrent interviews, memory growth) | **holds** (as a *real but partial and fragile* win — i.e. **not worth it here**) |
| **prompt-caching** (make Claude Code's automatic cache hit via stable cwd + 1h TTL) | Small (~a few hundred ms–1s on Q2..N; caches only the ~7.5k base prefix) | ~None (caching doesn't change what the user sees) | Small (Fix A: stable cwd) | Low (TTL vs think-time; per-call prefix determinism) | **partial** — literal cache_control approach refuted (CLI takes none); reframed stable-cwd win is real but modest |

## 4. Recommended plan

Do the work in two phases. **Phase 1 delivers the "feels like chat" win** and is where most of the value and effort lives. **Phase 2 is a near-free actual-latency top-up** that should ride in the same PR. Both stay behind the ADR 0007 seam (`get_brain()` in `api/app/interview.py:198-206`), preserve the scripted + codex paths, and honor single-worker uvicorn (`api/app/main.py:43-47`).

### Phase 1 — Perceived latency: pre-generation + streaming

The order matters: **pre-generation is the higher-value, lower-risk half** (it hides an entire round-trip and pre-warms Q1). Streaming layers on top of the same new SSE endpoint to render progressively. Ship pre-generation first, streaming second, both under one endpoint.

**1a. Backend — non-blocking answer + background pre-generation** (`api/app/routers/requests.py`, `api/app/schemas.py`)
- Rework `answer_interview` (`requests.py:162-174`) to record the `InterviewTurn`, set `pending_question=None`, commit, and **return immediately** with a state carrying a new `thinking: true` flag — it must NOT call `next_question`.
- Schedule generation in an **in-process** background task (thread / `ThreadPoolExecutor` / `BackgroundTasks`, the same in-process pattern the tick loop already uses per `main.py`). The task opens **its own DB session**, calls the SAME `get_brain().next_question(r)`, and writes `r.pending_question` — the exact value `current_question` would have written, so the seam, budget floor/ceiling, `{"done":true}` early-stop, scripted fallback, and `<request_data>` sandbox are all untouched (`api/app/agent_brain.py:70-116`).
- Make `GET /interview` (`current_question`, `requests.py:42-53`) a **cheap read**: if `pending_question` is set, return it; if generation is in flight, return `thinking:true` with no question. It must NOT trigger generation itself.
- Add one bool `thinking`/`generating` field to `InterviewState` (`api/app/schemas.py:206-214`).
- **Pre-warm Q1 at request create** (`requests.py:118-137`): schedule the same background `next_question` so by the time the user finishes the Describe step, `pending_question` is often already populated — this hides the one cold-start you cannot overlap with typing.
- **Guards** (the real risk surface): keep generate-once persistence (`requests.py:47-52`); add an in-flight guard keyed by `request_id + answered_count` so a late-finishing generation for turn N never overwrites turn N+1; give the client a max-poll ceiling so a 120s-timeout → scripted-fallback resolves the spinner instead of hanging.

**1b. Frontend — optimistic composer + short-poll/stream** (`apps/intake/src/app/submitter/interview.ts`, `packages/shared/src/lib/models.ts`)
- In `push()` (`interview.ts:715-731`): optimistically append the user's answer bubble to `turns()` immediately, show the Fabric typing row, POST once. On failure, **roll back the optimistic turn** and re-enable the composer (today's error path just clears `busy` — `interview.ts:683,729` — and must be hardened).
- Replace the one-shot constructor subscribe (`interview.ts:673-685`) with a short-poll (rxjs `timer`/`interval` ~1.5s) of `api.interview(id)` while `thinking`, until a question or `done` arrives. `api.service.ts` already has `interview()`/`answer()` — no new method needed for the poll variant.
- Add the `thinking` flag to `InterviewState` in `packages/shared/src/lib/models.ts`.

**1c. Backend — streaming SSE endpoint** (`api/app/agent_exec.py`, `api/app/agent_brain.py`, `api/app/routers/requests.py`)
- Add a generator variant of the claude path, e.g. `run_agent_stream(...)`, that spawns the CLI with `--output-format stream-json --verbose --include-partial-messages` (confirmed on the installed binary, Claude Code 2.1.187) and iterates `for line in proc.stdout:` yielding parsed `text_delta` strings, then a terminal `result` line carrying the full text. **Keep the existing batch `run_agent`** for `draft_spec`, the scripted fallback, and codex. Preserve `start_new_session=True` + `killpg` whole-tree kill on the 120s bound (`agent_exec.py:83-104`). See [Anthropic streaming docs](https://platform.claude.com/docs/en/docs/build-with-claude/streaming) and [stream-json headless](https://code.claude.com/docs/en/agent-sdk).
- **codex cannot token-stream** — it returns its final message via `--output-last-message` file, not stdout (`agent_exec.py:141-160`). The seam MUST branch: stream for claude (the dev default per `launch.json`), fall back to the batch/pre-generation path for codex and scripted.
- Add a **new `async def` SSE endpoint** returning `fastapi.responses.StreamingResponse(gen(), media_type="text/event-stream")`, formatting each delta as `data: {json}\n\n`. It MUST run the blocking subprocess reader **off the event loop** (`anyio.to_thread` / a queue) so the single uvicorn worker's event loop is never frozen — `main.py:60` already uses `to_thread` for exactly this reason. Persist `r.pending_question` **exactly once** on stream completion and have the terminal SSE event carry the same `InterviewState` the batch endpoint returns, so the client reconciles. Keep the batch `GET` as the reconnect/replay source of truth. See [FastAPI SSE](https://fastapi.tiangolo.com/tutorial/server-sent-events/) and [MDN EventSource](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events).

**1d. The prompt/contract change streaming needs** (`api/app/agent_brain.py:86-102`) — *this is the real design cost of streaming.*
- The prompt today asks for ONLY-JSON `{"question":...,"options":...}`. Streaming that raw shows the user `{"question": "Wh…` — broken, not fast. Change the contract so the model emits **human-readable question text first** (e.g. wrapped `<q>…</q>`), then an options JSON tail. The streamed `<q>` text drives the live bubble; the JSON tail is parsed for options/final on completion.
- This touches the ADR 0007 seam parsing: the replacement MUST still return an `interview.Question` (`question/sub/options/final`) or `None`, still degrade to `super().next_question(req)` with re-stamped `final` on bad/empty/timeout output (`agent_brain.py:104-116`), still honor floor/ceiling/`{"done":true}`, still keep the `<request_data>` injection defense and throwaway-cwd/attachment handling.

**1e. Frontend — stream consumer + a11y** (`packages/shared/src/lib/api.service.ts`, `apps/intake/src/app/submitter/interview.ts`)
- Add an `EventSource` (or `fetch`-ReadableStream) consumer in `api.service.ts` (today only one-shot `http.get`/`post`, `api.service.ts:63-68`). Append `text_delta` into a signal; on the terminal event set `st()` to the reconciled `InterviewState`.
- The `busy()` typing row (`interview.ts:76-83`) becomes a live-updating bubble.
- **Accessibility:** the `aria-live` region / `liveQuestion` computed (`interview.ts:33,665-671`) must announce the **final question once**, not every token, or screen readers get spammed. Respect the existing `prefers-reduced-motion` handling.
- **Dev proxy / prod nginx:** disable response buffering (`X-Accel-Buffering: no`) on the SSE route in `proxy.conf.json` and nginx, or deltas arrive batched and silently defeat the win.

**Phase 1 tests:** the interview tests inject a fake executor at the `agent_exec` seam (`api/tests/test_agent_brain_attachments.py`) — the fake must now also support a generator variant; add a test asserting the stream **degrades to the scripted brain on garbage/timeout**, and one asserting the background generation writes `pending_question` exactly once and never overwrites the wrong turn. `make verify` stays hermetic (scripted default makes no network calls).

### Phase 2 — REJECTED: split-model (Haiku for questions)

**Not doing this.** Haiku 4.5 would cut the actual round-trip ~30-50% and is ~3x cheaper, and the verifier confirmed the latency claim *holds* — but the decision is **question quality over speed**. Haiku risks weaker, less-incisive follow-ups exactly on the deep "new app" interviews (up to 10 questions walking the design tree) where sharp questions matter most, and Sonnet 5's question quality is already proven in live testing. Keep `next_question` on Sonnet 5.

**Kept only as an emergency knob, not a default:** the mechanism (a `FACTORY_INTERVIEW_MODEL` env knob threaded `run_agent` → `_run_claude_cli` → `_claude_cmd`, defaulting to `CLAUDE_MODEL`) is cheap to build and would let us A/B Haiku's *actual* question quality later if we ever want to revisit — but it ships off by default (Sonnet).

**Quality-preserving speed lever (deferred, not in this plan):** disable Sonnet's *extended thinking* for the narrow question step (`thinking:disabled` / `effort:low`) — this keeps Sonnet's judgment and language while cutting the round-trip, since picking the next question isn't deep-reasoning work. Catch: that knob is **not exposed by the `claude -p` CLI**; it requires the deferred direct-API path (§5). Revisit only if Phase 1 proves insufficient.

### Sequencing summary

1. **Phase 1a + 1b** (pre-generation, non-blocking composer) — biggest perceived win, lowest risk, no model/quality change. Ship first.
2. **Phase 1c-1e** (streaming) — the "typing back" polish on top of the same SSE endpoint. Ship once pre-generation is proven.
3. **(Deferred)** direct-API path → then the `thinking:disabled` speed lever, *only if* Phase 1 isn't enough. No Haiku.

## 5. Explicitly rejected

- **warm-worker (persistent `ClaudeSDKClient` / warm pool) — REJECTED.** A real but *partial and fragile* win: it amortizes only CLI cold-start (~seconds), not the 60-90s model round-trip that dominates. Worse, it fights three hard constraints — the SDK is async-only vs this repo's sync SQLAlchemy routers (needs an event-loop-on-background-thread bridge); a single shared client [cannot isolate concurrent interviews by session](https://github.com/anthropics/claude-agent-sdk-python/issues/560), forcing a serializing lock or a memory-multiplying pool under single-worker uvicorn; and it breaks the `FACTORY_CLI` seam because [the SDK is "optimized for single-user CLI usage, not multi-tenant servers"](https://github.com/anthropics/claude-agent-sdk-python/issues/333) and has no codex equivalent. It also doesn't deliver the "feels like chat" outcome without *also* adding streaming — at which point the direct API is a cleaner streaming source. The predictive-ui approach (Phase 1) captures the cold-start-hiding benefit by overlapping generation with think-time, without any of this machinery.

- **prompt-caching as a latency lever — REJECTED (kept only as an optional cost tweak).** The literal "set `cache_control` breakpoints" is **not implementable through the `claude -p` CLI** — the repo has no `anthropic` SDK and the CLI takes no cache flags. The in-scope version (give Claude Code a stable cwd so its *automatic* server-side cache hits across questions, plus `ENABLE_PROMPT_CACHING_1H=1` to survive human think-time) is real but caches only the ~7.5k-token base prefix and saves a few hundred ms–1s per question — it does **not** move the 60-90s round-trip and does **not** change what the user sees. It is a cost lever first, a latency lever a distant second. If Phase 2's stable per-interview scratch dir happens to make the cache hit, that's a free bonus, but it is not worth engineering for its own sake.

- **direct-api-not-cli as the *headline* fix — DEFERRED, not the recommended entry point.** Technically sound and a genuine safety upgrade (a tool-less Messages call makes the empty-cwd sandbox, `--disallowed-tools`, and `--max-turns` moot), and it unlocks native `effort:low`/`thinking:disabled`, structured JSON output, and clean prompt caching. **But its headline oversells:** dropping the subprocess deletes only the cold-start tranche (seconds), not the 60-120s — that number is dominated by the model round-trip, which the transport swap does not touch, and the big perceived win still requires the *same* net-new SSE infrastructure Phase 1 builds. It also adds per-token billing, a shared org rate-limit pool, and container auth (`ANTHROPIC_API_KEY`) that must fail closed to scripted. Recommendation: **reach for it only if** Phase 1 + Phase 2 prove insufficient and you want `effort:low`/`thinking:disabled` + `cache_control` on the static preamble — then port `run_agent`'s claude branch to `messages.stream()` behind a new `FACTORY_CLI=api` seam value. It is the right *long-term* direction (ADR 0007 already anticipates a thin LLM adapter), but it is a bigger, riskier change than the recommended plan and should not be the first move.
