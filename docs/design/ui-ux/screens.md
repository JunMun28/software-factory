## 1. S0 — Login / SSO landing (Microsoft SSO front door)

**Primary user:** submitter

**Purpose.** The shared, role-agnostic front door to the entire Software Factory web app. Its only job is to authenticate the person via Microsoft SSO (Entra ID) and then route them — Submitters into the Intake shell (New Request), Admins into the Control center — based on the role resolved from the validated Entra session. It must feel minimal, fast, and trustworthy: this is the first impression and the place a non-technical Submitter decides "yes, this is my company's tool, I can sign in safely." Exactly one primary action (Sign in with Microsoft); no password field, no sign-up, no marketing. It sets the brand tone (calm neutral+Micron purple, Inter, restraint) so the product reads as one coherent system from the very first frame. Because login precedes the role fork, it is rendered in the SHARED neutral register; the foundation's "Submitter = light theme" default is honored here so the SSO handoff reads warm and approachable rather than mission-control-dark.

### Layout
Single full-viewport centered composition on a calm light canvas (bg-canvas #FBFBFC) — no sidebar, no top header, no Cmd+K (the Submitter shell has no chrome, and login precedes the shell fork entirely). One vertically-centered column, max-width ~400px, optically centered (slightly above true center, ~46% from top) so it never feels bottom-heavy.

Regions, top→bottom inside the centered column:
1. BRAND LOCKUP (top): a small de-iconified Factory mark (monochrome line glyph, ~32px, inheriting currentColor in text-primary) + wordmark "Software Factory" in h2/600 with negative tracking. A one-line tagline beneath in text-secondary/caption: "The front door for software requests." This is the entire brand statement — no hero, no illustration.
2. AUTH CARD (center, the focal object): a single white bg-surface card, radius-lg (12px), elev-1 (1px inset hairline border-subtle + soft light-theme shadow 0 1px 3px rgba(16,24,40,.10)), padding 32/24. Contains, stacked: a short context line ("Sign in to file or track a request") in body-lg/text-primary; THE primary action — a full-width "Sign in with Microsoft" button (44px tall, radius-sm, accent #BD03F7 fill, white label, the 4-square Microsoft logo at left in its official quadrant colors as the ONE sanctioned exception to the monochrome rule, since it is a recognized trust signal); below the button, a single muted reassurance line in caption/text-secondary: "Uses your company Microsoft account — no new password."
3. FOOTER META (bottom of column, generous 48px gap above): a hairline-separated row in micro/text-tertiary — a single-sentence trust/help line ("Trouble signing in? Contact IT.") and, far-right, a small text-only theme toggle affordance (sun/moon) for users who prefer dark. No legal wall, no nav links.

The whole composition sits on whitespace; structure is felt via the single card edge and the 8px rhythm, not via boxes or rules. On desktop the column is centered in a wide empty canvas; the emptiness is intentional restraint, not an unfinished screen.

### Key components
- Brand lockup — Factory line-glyph mark + 'Software Factory' wordmark (h2/600, negative tracking) + one-line tagline; the entire brand statement, no hero art
- Auth card — single bg-surface card (radius-lg, elev-1, 32px padding) that is the one focal object on the canvas; the 'card edge as structure' pattern from the foundation
- Primary SSO action — full-width 'Sign in with Microsoft' button (accent fill, 44px, radius-sm) with the official 4-color Microsoft quadrant glyph as the sole sanctioned color exception; THE one primary action, no secondary button competing with it
- Reassurance microcopy — 'Uses your company Microsoft account — no new password' (caption/text-secondary), the anti-friction trust line that answers a non-technical Submitter's only real question
- Focus ring — global :focus-visible 2px Micron purple focus-ring at 2px offset on the SSO button (and theme toggle), keyboard-visible from first load
- Three-state discipline affordance — the SSO button is the single surface that carries idle / loading-spinner / error states inline, per the foundation's skeleton/spinner/optimistic doctrine
- Footer meta row — micro/text-tertiary trust+help line ('Trouble signing in? Contact IT.') and an unobtrusive light/dark theme toggle (honoring prefers-color-scheme default, persisting choice)
- Inline error banner — recoverable, specifically-worded error surface that names the failed step (denied / popup-blocked / not-provisioned / network), reusing the foundation's 'specific recoverable error' rule, never a generic 'something went wrong'

### States
- **Idle / default (populated)** — Brand lockup, auth card, and footer fully rendered on the light canvas. SSO button at rest in accent fill with the Microsoft glyph and label 'Sign in with Microsoft'. Reassurance line visible. This is the canonical first frame — calm, complete, one obvious action.
- **Initial load / session-check** — Brief moment on app open while Auth checks for an existing valid Entra session (silent SSO). Render the full layout immediately with the card present but the SSO button showing a quiet inline spinner and label 'Checking your sign-in…', so the page never flashes empty. Resolves either to Idle (no session) or skips straight to redirect (valid session) — most returning users never consciously see Idle.
- **Authenticating (loading)** — User clicked Sign in. The button enters loading: label swaps to 'Opening Microsoft…', a compositor-only spinner (transform/opacity) replaces the glyph, button is disabled (aria-busy=true), focus retained on the button. The Microsoft popup/redirect is now driving; the page behind stays calm and unchanged (no skeleton, no whole-page spinner — three-state discipline says spinner only on the active surface).
- **Redirecting (post-auth role fork)** — Auth succeeded; the validated Entra session resolved a role. The button reads 'Signing you in…' and the page holds steady for the sub-second redirect — Submitter → S1 New Request, Admin → Control center board. No content swap on this screen; the route change carries the transition (slow 220 cross-fade), so the handoff feels like one motion, not two pages.
- **Error — access denied / cancelled** — User closed the Microsoft popup or denied consent. Button returns to idle (re-enabled, label restored). A specific inline banner appears above the button in a quiet style (not loud red — this is recoverable, not Needs-human): 'Sign-in was cancelled. Try again when you're ready.' Focus moves to the banner then back to the button. No red alarm color — amber/red are reserved for gates/Needs-human only.
- **Error — not provisioned / no role** — Auth succeeded but the account has no Submitter or Admin role (not in the org's allowed group). Inline banner, still recoverable-toned: 'Your account isn't set up for the Software Factory yet. Contact IT to get access.' SSO button stays available but secondary; the help line is emphasized. This is the one place the screen explains the role gate without exposing internal vocabulary.
- **Error — popup blocked** — Browser blocked the SSO popup. Specific, actionable banner: 'Your browser blocked the sign-in window. Allow pop-ups for this site, or use the link below.' offers a fallback full-page-redirect link (no popup) so the user is never stuck — naming the exact failed step, per the foundation's error rule.
- **Error — network / IdP unreachable** — Entra/network failed. Banner: 'Couldn't reach Microsoft sign-in. Check your connection and try again.' Button re-enabled for retry. Still no loud color; the tone stays reassuring because nothing is broken on the user's side.
- **Already signed in (returning, valid session)** — Silent SSO finds a live session on load; the screen is effectively transient — it renders the load/session-check frame then immediately redirects to the role-correct destination without requiring a click. A returning user perceives the app 'just opening,' which is the intended felt-speed.
- **Dark theme variant** — If the user toggled dark (or prefers-color-scheme:dark) the same composition renders in the Control-center dark register (bg-canvas #0B0C0E, bg-surface #141518, text-primary #F2F3F5, accent #BD03F7, depth via 1px inset hairline). Identical structure; theme is a pure token swap. This keeps anyone who opts into dark from a jarring flash; light is the default for both Submitters and Admins.

### Interactions
- Click / Enter / Space on the SSO button triggers Microsoft sign-in (popup, with full-page redirect fallback). The button is the default focused control on load, so a keyboard user can authenticate with a single Enter — keyboard parity for the one primary action.
- Tab order is trivially short: SSO button → fallback/help link (only if an error exposed it) → theme toggle. :focus-visible Micron purple ring is present from first paint; Tab is never required before the primary action is reachable.
- Optimistic-feel loading: the moment the button is clicked it transitions to the loading label/spinner within instant(80ms) so the click feels acknowledged even before Microsoft responds — felt speed without a sync engine.
- On a thrown error the button rolls back to idle and re-enables; the inline banner is announced via aria-live=polite and focus is managed so a keyboard/AT user lands on the explanation, not silently back at a re-enabled button.
- Theme toggle flips light/dark instantly (token swap, no reload), respects and overrides prefers-color-scheme, and persists the choice (localStorage) so the next visit honors it; honors prefers-reduced-motion by cross-fading rather than animating.
- No Cmd+K, no G-nav, no single-key verbs here — by design. The Admin keyboard grammar belongs to the post-login shell; login deliberately exposes none of it, keeping the front door uncluttered for non-technical Submitters.
- Redirect after success is automatic and role-driven (no 'continue' click) — the screen never asks the user to choose Submitter vs Admin; the validated session decides, so the door opens to exactly the right room.

### Borrows from Linear / Slack
- Linear — radical restraint: a near-empty canvas with one card and one action, trusting whitespace and a single hairline card edge to carry structure ('structure felt, not seen'). Linear's login is famously minimal; this mirrors that confidence.
- Linear — quiet neutral+Micron purple register and tight Inter typography with negative tracking on the wordmark, establishing 'this is one coherent, engineering-grade product' from frame one.
- Linear — global :focus-visible Micron purple focus ring and keyboard-reachable primary action, even on a screen with almost nothing to navigate, so the keyboard-first discipline is consistent end to end.
- Linear/foundation — three-state discipline (idle / spinner-on-the-active-surface / specific recoverable error), applied to the single SSO button rather than ever flashing the whole page.
- Foundation — 'quiet by default, loud only when it matters': errors here are recoverable and rendered in a calm tone, explicitly NOT amber/red, preserving those two colors exclusively for gates and Needs-human deeper in the product.
- Foundation — token-swap theming (light for Submitter-facing, dark for Admin/Control-center), demonstrated on the very first screen so the two faces of one system are coherent before the role fork.
- Slack — the trust-forward, low-friction SSO front door pattern (a single big branded 'Sign in with [provider]' button + reassurance microcopy, no password field) that non-technical users already recognize and trust.
- Foundation — teaching/reassuring microcopy in the spirit of the empty-state voice ('no new password', 'Contact IT'), guiding a novice Submitter without ever exposing Control-center vocabulary or GitHub.

## 2. S1 — New Request form (Submitter)

**Primary user:** submitter

**Purpose.** The Factory's front door: let a non-technical Submitter file a Request in under a minute by first choosing its type (bug fix / enhancement / new app), which reveals only the conditional fields that type needs. It captures structured-enough intent — title, plain-language description, the existing App (for bug/enhancement), urgency, optional attachment — to feed the Intake interview and Draft-spec generation that follow. North star: quick, low-friction, one-and-done; identity is taken from the validated Microsoft SSO session so the Submitter never types who they are and never sees GitHub, repo names, or Control-center vocabulary.

### Layout
Submitter shell (the forked light-theme, mobile-first experience — NO sidebar, NO Cmd+K). Minimal top bar spanning full width: product wordmark left, the two Submitter destinations as quiet text links center-left ("New Request" active, "My Requests"), signed-in identity chip far right (avatar + name from Entra session, dropdown for sign-out). Below it a single centered content column, max ~720px, generous top padding, page background bg-canvas (#FBFBFC), the form on a white bg-surface card with hairline border-subtle and lg (12px) radius — though on the smallest breakpoints the card edge dissolves to full-bleed.

The form is a single vertical flow on the 8px rhythm, top→bottom:
(1) Header zone — h1 "New Request" + one caption line of reassurance ("Tell us what you need in your own words — we'll ask a couple of quick questions after.").
(2) Type selector — the decision that gates everything else: a labeled group of three large segmented radio cards (Bug fix · Enhancement · New app), each with a small monochrome line glyph, a one-line plain-language helper, full-width on a row, 12px gutter. This is the only always-visible required input; nothing below it renders until a type is chosen.
(3) Conditional field region — animates in below the selector once a type is picked. Shared fields render first (Title, Description); then type-specific fields; then the universal Urgency control and optional Attachment. The region is the same column width so the eye-path is one ruler-straight line of fields, each with a 14px-equivalent label (rendered at body-lg/16 for Submitter), helper caption beneath the label, input at full width.
(4) Action footer — sticky to the bottom of the viewport on mobile, inline at the end of the column on desktop: a primary Submit button (Micron purple accent, right/leading-aligned) labeled "Continue to questions →" (signals the interview comes next, not a dead-end submit), a secondary "Save as draft" ghost link, and a micro caption "Signed in as Maya Chen · maya.chen@northwind.com" so identity provenance is visible but unobtrusive.

Field inventory by type:
• Shared (all types): Title (single line), Description (multiline, plain-language).
• Bug fix: App dropdown (required) + "What happens now" + "What you expected" (two short multilines, side-by-side on desktop, stacked on mobile).
• Enhancement: App dropdown (required) + "What's the goal / who's it for" (short multiline).
• New app: "What problem does it solve" + "Who will use it" (two short multilines) — NO App dropdown (the repo doesn't exist yet).
• Universal tail: Urgency (3-stop segmented control: Low · Normal · High, defaulting Normal) + Attachment (optional drop/upload zone for a screenshot or paste-a-link field).

### Key components
- Submitter shell top bar (wordmark · New Request / My Requests links · Entra identity chip) — the minimal forked shell, no sidebar, no Cmd+K
- Type selector as three segmented radio cards (Bug fix · Enhancement · New app) — borrows Linear's segmented type pickers; the only always-visible required input; each card carries a monochrome line glyph + one-line helper
- Conditional field region — fields that animate in keyed to the chosen type; reveal not replace, so changing type re-renders only the delta
- App dropdown (existing apps) — fed by the App registry; a searchable/typeahead select showing friendly app names ONLY (never repo names or owners); required for bug/enhancement, absent for new app
- Title input + Description textarea (shared) — body-lg/16 plain-language fields, helper captions, auto-growing description
- Urgency segmented control (Low · Normal · High, default Normal) — restrained 3-stop, not a slider, maps to Request.urgency
- Optional Attachment zone — drag-drop / click-to-upload for a screenshot plus a paste-a-link field; clearly optional
- Primary action 'Continue to questions →' — Micron purple accent button that hands off to S2 Intake interview; signals the interview is next, not a terminal submit
- Identity provenance caption ('Signed in as …') — the anti-retype, taken from validated Entra session
- Three-state discipline scaffolding — skeleton on App-dropdown load, inline field validation, specific recoverable submit-error banner, teaching empty/first-run header copy

### States
- **First load / type not yet chosen** — Header + the three type cards rendered; everything below the selector is absent (not greyed — simply not rendered yet), keeping the initial screen radically simple. A faint helper under the selector reads 'Pick a type to get started.' Submit is not shown. This is the calm front-door default.
- **App-dropdown loading (skeleton)** — When Bug fix or Enhancement is chosen, the App dropdown mounts; while the App registry list resolves it shows a skeleton shimmer in the field shape (structure, not a spinner) and is non-interactive. The rest of the form is fully usable meanwhile.
- **App registry empty** — If the registry returns zero apps (edge case for a fresh deployment), the dropdown shows a teaching empty state inside the menu — 'No apps registered yet. Choose New app instead, or ask an admin to add one.' — rather than an empty list. Field is marked so it can't be left in an unresolvable required state.
- **Populated — Bug fix** — App dropdown (required) selected, Title + Description filled, 'What happens now' / 'What you expected' both filled, Urgency set, optional attachment thumbnail present. Submit enabled.
- **Populated — Enhancement** — App dropdown + Title + Description + 'goal / who for', Urgency. No bug-specific fields. Submit enabled.
- **Populated — New app** — No App dropdown; 'What problem does it solve' + 'Who will use it' + Title + Description + Urgency. Submit enabled. Copy subtly signals a new repo/app will be created on approval (without using the word repo).
- **Type switched after partial entry** — User changes the type segment after filling shared fields: Title/Description/Urgency persist; only type-specific fields swap (the delta animates out/in). A quiet inline note appears if previously-entered type-specific data will be dropped, with the change being non-destructive to shared fields.
- **Inline validation error** — On blur or submit attempt, a required-but-empty field (Title, Description, or App for bug/enhancement) gets a border-strong red-tinted edge + a caption-sized error message naming the fix ('Add a short title so admins can scan it'). Focus moves to the first invalid field; error region is announced. Submit stays enabled (errors surface on press), never silently dead.
- **Submitting (optimistic + activity)** — On Submit the button enters a loading affordance (label → 'Saving…' with an inline spinner, control disabled to prevent double-submit). The persist-first contract means the Request is saved before any GitHub Issue; on success it hands off to S2 Intake interview in the same window. Felt as instant — the wait is removed by transitioning straight into the interview view.
- **Submit error (recoverable)** — If save fails, a specific recoverable error banner appears above the footer naming the failed step and offering Retry ('Couldn't save your request. Nothing was lost — try again.'). Entered data is preserved; the button returns to actionable. Never a generic 'Something went wrong.'
- **Saved-as-draft confirmation** — Choosing 'Save as draft' persists the in-progress Request (status Draft) and shows a quiet inline toast ('Draft saved — pick it back up from My Requests'), no navigation forced.
- **Unauthorized / session expired** — If the Entra session has lapsed mid-form, a non-destructive overlay prompts re-authentication and returns the user to the same filled form afterward (data preserved client-side), respecting the shared unauthorized state.

### Interactions
- Choosing a type card reveals the conditional region with a base/160ms transform+opacity entrance (slide-down + fade), one field group at a time on the 8px rhythm; honors prefers-reduced-motion by cross-fading instead of sliding.
- Type cards are a true radio group: arrow-key Left/Right (and Up/Down) move selection, Space/Enter selects, only one active; the active card carries an Micron purple accent border + filled glyph and a visible focus ring.
- App dropdown opens a searchable menu (typeahead filters the registry list); keyboard: type to filter, Up/Down to move, Enter to choose, Esc to close; menu uses bg-surface-raised popover elevation.
- Description and the type-specific multilines auto-grow with content up to a sensible cap, then scroll — no fixed tiny box, no manual resize handle needed.
- Urgency segmented control is a 3-stop radio: click or arrow-keys to set Low/Normal/High; Normal pre-selected so it's never an empty required field.
- Attachment zone accepts drag-drop of an image, click-to-browse, or a pasted URL in the adjacent link field; a chosen file shows a small thumbnail + filename + remove (×); paste of an image from clipboard is accepted.
- Submit ('Continue to questions →'): validates, then optimistically transitions into the Intake interview (S2) in the same window; disabled+spinner during save to block double-submit. Cmd/Ctrl+Enter from any field also submits (the one power-keystroke afforded to Submitters).
- Switching type after entering type-specific data prompts a quiet inline confirm only for the data that would be dropped; shared fields are untouched.
- Tab order is strictly top→bottom (type → shared → type-specific → urgency → attachment → submit); the sticky mobile footer never traps or obscures focus (focus scrolls into view above it).
- Identity chip dropdown offers Sign out only; everything else about identity is read-only and auto-filled.

### Borrows from Linear / Slack
- Linear — segmented type/property pickers: choosing a Request type up front is the same move as Linear setting issue type/status, and it conditionally gates the rest of the form (progressive disclosure, not a wall of fields).
- Linear — restraint & quiet-by-default: a calm neutral+Micron purple light theme, hairline borders, whitespace carrying hierarchy; the Micron purple accent reserved for the single primary action; no loud color anywhere on this Submitter screen (amber/red are Admin-only signals and don't appear here).
- Linear — status-TYPE-by-shape thinking applied to type cards: each type carries a monochrome line glyph so the choice reads by shape, not color alone (WCAG 1.4.1), consistent with the product's glyph language.
- Linear — keyboard parity even for a guided form: full radio-group arrow navigation, typeahead dropdown, Cmd/Ctrl+Enter to submit — keyboard-optional but available.
- Linear — three-state discipline: skeleton for the App-dropdown load, teaching empty state when the registry is empty, and specific recoverable errors that name the failed step rather than a generic toast.
- Slack — channel-as-app mental model surfaced gently: the App dropdown picks the Subject (the per-app channel/feed the Request will later live under), but shown to the Submitter purely as a friendly app name, never the repo/feed internals.
- Shared design system — one auth, two faces: this is the Submitter face (light-friendly, 16px body-lg, mobile-first, plain-stage vocabulary) of the same token set + shell that powers the Admin Control center; identity auto-filled from the validated Entra session.
- Shared — 'grounded' philosophy upstream: capturing crisp type-specific fields here is what later lets the Draft spec tag each line (from: Q-x)/(ASSUMPTION); the form is the provenance source even though tags appear on the Admin side.

## 3. Intake interview (AI chat) — S2

**Primary user:** submitter

**Purpose.** Right after a Submitter hits Submit on the New Request form (S1), this screen runs a short, warm, real-time AI follow-up chat in the SAME window — the Intake interview. It reads the just-submitted Request and asks ~3–4 adaptive clarifying questions to fill gaps while the Submitter is still present and the context is fresh, then stops (cap or "enough"), writes the Draft spec, and hands off to the Submission confirmation (S3). Its job is to feel like a quick, friendly conversation that visibly converges — never an interrogation, never an open-ended chatbot. It is the one Submitter surface where genuine real-time token streaming is legitimate (this Stage-1 brain runs as direct in-app LLM calls with the user waiting, per PRD line 75), which is categorically different from the Control-center Progress feed where ADR-0004 forbids streaming. The Submitter never sees GitHub, the Draft-spec internals, the word "spec," or any Control-center vocabulary.

## 4. S3 — Submission confirmation

**Primary user:** submitter

**Purpose.** Close the Intake loop with a calm, unambiguous "we've got it" the instant the AI interview ends and the Draft spec is generated. This is the handoff moment: the Submitter has just answered ~3–4 follow-ups, the enriched Request is persisted and a GitHub Issue is being opened in the background, and they need three things and nothing more — (1) reassurance the Request truly landed (with a reference they can cite), (2) an honest, plain-language preview of what happens next (an Admin reviews the AI's Draft spec, then approves or sends it back — no GitHub, no spec internals), and (3) a single obvious way to track it. It primes the plain-stage tracker (Submitted → Spec drafted → Approved → Building → In review → Deployed, or Needs a human) so My Requests / Request detail feel familiar later. Deliberately a dead-end-and-launchpad: the work is done, so the screen does almost nothing — it resists the temptation to show the Draft spec (Admin-only, ADR-0003), a progress firehose (ADR-0004), or any Control-center vocabulary. Structure shown, not a centered spinner — the three-state discipline. No layout shift when real rows replace skeletons.
- **Populated — steady (nothing needs me)** — The common, calm case. Needs-your-input band is absent. A reverse-chron stack of row-cards, each with a quiet plain-stage pill (Submitted / Spec drafted / Approved / Building / In review). Entirely neutral+Micron purple register; no amber, no red. Reads as 'everything's moving, relax.'
- **Populated — one or more Needs-your-input** — The hero state of this screen. A 'NEEDS YOUR INPUT' band sits pinned above the main list holding each sent-back Request with its amber pill + 'Respond' affordance and a one-line preview of why ('Admin asked: which teams need access?'). The rest of the list renders normally below. Exactly the rows that need action are loud; the rest stay quiet.
- **Active vs All** — Active (default) hides Deployed and Cancelled to keep the list to live work. Toggling All reveals them, appended with check/strike glyphs and muted styling (Cancelled uses strike glyph + tertiary text + a small 'Cancelled by admin' caption). Toggle state is the only filter; persists for the session.
- **Empty — no requests yet** — First-time Submitter. Centered teaching empty state: 'You haven't filed any requests yet.' + subline 'File a request and an AI will ask a couple of quick questions, then it's on its way.' + primary 'New Request' button. No skeleton, no fake rows.
- **Empty — All filtered, none active** — Edge: user has only Deployed/Cancelled requests and is on Active. Soft message: 'Nothing in progress. Switch to All to see your finished requests.' with an inline 'View all' affordance — never a dead blank.
- **Optimistic refresh (poll)** — On the gentle background poll (this is a Submitter screen — slower cadence than the Admin 3–5s board; no auto-animate spam), a row whose stage advanced updates its pill in place with a fast cross-fade; a newly sent-back Request lifts into the Needs-your-input band with a base-duration ease-in-out move. Never a whole-list refresh flash; only changed rows transition. prefers-reduced-motion swaps the move for an opacity cross-fade.
- **Just-acted / returned from S5b** — After the Submitter responds to a send-back in S5b and returns here, that Request leaves the Needs-your-input band and re-enters the main list with a 'Resubmitted · Spec drafting' pill and a brief confirming toast ('Thanks — back with the team for review'). If that empties the band, the band animates away (collapse), removing the amber from the page entirely.
- **Error — list failed to load** — Specific, recoverable: 'Couldn't load your requests. This is on our side, not yours.' + a 'Try again' button that re-fetches. Top bar and New Request stay functional so the user is never trapped.
- **Unauthorized / session expired** — Session no longer valid (Entra expiry). Full-screen friendly redirect-to-login state ('Your session timed out — sign back in to see your requests') rather than a raw 401; routes to S0. Ownership scoping means a Submitter can never be shown another user's rows even transiently.
- **Single long title / overflow** — A long Request title truncates to one line with ellipsis on the row; type chip, pill, and timestamp never get pushed off — the pill (the load-bearing status) is pinned right and protected. Full title shows on hover (desktop) and is fully visible in S5 detail.

### Interactions
- Click/tap a row → opens S5 Request detail (Submitter, plain) — full-page route on mobile, same on desktop (no in-place side-panel; that's an Admin pattern). Whole row is the hit target, ≥48px tall.
- Click 'Respond' on a Needs-your-input row (or click the row) → routes to S5b Respond-to-send-back, the first-class flow that surfaces the admin's note and lets the user add info / short re-interview.
- 'New Request' (top bar) → S1 New Request form. Always present, even in empty/error states, so the front door is never blocked.
- Active · All toggle → instant client-side filter over already-loaded local list (no network), state persists for session; the only filter control on the page.
- Keyboard (desktop, secondary not primary — Submitters are mostly mouse/touch): Tab moves through rows in DOM order; Enter/Space on a focused row opens S5; the toggle is a proper radio/segmented control reachable by Tab + arrow keys. No G-nav, no single-key verbs — that grammar is Admin-only.
- Hover (desktop) → row raises one elevation step (border-strong edge + faint lift), full title tooltip on truncated titles; touch has no hover dependency — everything reachable by tap.
- Background poll updates pills/band in place; an in-app S6 notice arriving while on this screen drops a dismissible toast at the top of the list and, if it's an approval/send-back, updates the relevant row simultaneously.
- Pull-to-refresh on mobile (native gesture) as an explicit manual refresh on top of the gentle auto-poll.

### Borrows from Linear / Slack
- Linear — Triage queue, repurposed: the Needs-your-input band IS the Submitter-side Triage inbox — the small set of items that actually require a decision, lifted out of the calm stream above the rest. Same 'act on the few, ignore the many' shape, stripped to one verb (Respond).
- Linear — Status-type glyph (shape=type, color=status): the plain-stage pills are the Submitter translation layer over the same dotted/ring/check/strike glyph set, so status reads color-blind even in the light theme.
- Linear — Attention badge as the one loud thing on an otherwise calm row: 'Needs your input' (amber) is the Submitter rename of the Admin 'Approve spec' badge — quiet by default, loud only when it's the user's move.
- Linear — restraint / density-with-grouping: a single scannable row (title · type · app · stage · time), AND-only Active/All toggle, no user-authored views, no WIP, no Cmd+K for this audience — deliberately under-built for one-and-done Submitters.
- Linear — three-state discipline + teaching empty states ('You haven't filed any requests yet…' / 'No specs waiting on you'-style reassurance) and specific recoverable errors.
- Slack — clear-to-zero vs ambient distinction, inverted for the Submitter: unlike the Admin's never-done ambient feed, this list IS finishable — the Needs-your-input band clears to zero and the 'That's everything' end cap states completion. The one place the Submitter side intentionally diverges toward 'inbox you can empty.'
- Shared design foundation — Submitter shell (16px body-lg, light theme, mobile-first, plain-stage vocabulary, no Control-center words), one auth + one token set forked at login.

### Responsive
Mobile-first (this is the primary Submitter context — US37 'quick on mobile and desktop'). Phone (≤480px): full-bleed single column, 16px side padding, row-cards stack the title on line 1 and a meta line (type chip · stage pill · time) on line 2 to stay tap-friendly; New Request collapses to an icon+label or a sticky bottom-right FAB if the top bar crowds; Needs-your-input band spans full width at the very top; native pull-to-refresh. Tablet (481–768px): same single column, comfortable line-length, rows can return to one-line layout. Desktop (≥769px): centered ~760px reading column with wide neutral gutters (no stretching to full width — this is reading-grade, not a dense board); hover affordances and keyboard focus enabled; the page never adopts the Admin sidebar. ### Accessibility
WCAG 2.2 AA throughout. Each row is a single semantic link/button with an accessible name composed of title + type + plain stage + 'updated Xh ago' (and 'needs your response' when applicable) so a screen-reader user hears the full state without visual scanning. The Needs-your-input band is a labelled region (aria-labelledby the 'NEEDS YOUR INPUT' heading) and announced; new sent-back items use a polite aria-live update on poll, not assertive (no hijacking). Targets ≥48px tall on touch, ≥24px effective everywhere (2.5.8). prefers-reduced-motion swaps pill cross-fades and band lift/collapse for instant opacity changes. All motion is feedback, not decoration; the gentle poll never animates unchanged rows.

## 6. Request detail (Submitter) + respond-to-send-back

**Primary user:** submitter

**Purpose.** The Submitter's private, plain-language home for ONE Request: read its status as a calm plain-stage timeline (Submitted → Spec drafted → Approved → Building → In review → Deployed, or the Needs-a-human side state), and — when an Admin sends it back — read the admin's note in full and respond by adding/correcting info or re-answering a short interview, all without ever touching GitHub, the internal Draft spec, repos, PRs, or any Control-center vocabulary. It is the Submitter end of the SentBack → PendingApproval loop (PRD hardening #7) and the first-class home of US15/S5b. Respond-to-send-back is NOT a hidden mode buried in the detail page; when a send-back is active it is the loudest, top-most thing on the screen.

### Layout
Submitter shell (S5/S5b): minimal top bar only — product wordmark left, "My Requests" back-link, user avatar/name (from Entra) right. NO sidebar, NO Cmd+K, NO board. Single centered reading column, max ~720px, generous margins, light theme (#FBFBFC canvas, #FFFFFF cards), body-lg 16px, mobile-first.

Vertical stack top→bottom:
1. Breadcrumb / back affordance: "← My Requests" (returns to S4). Small caption.
2. Request header block: the Submitter-given title as h1 (e.g. "Export invoices to CSV"), under it a metadata line of caption chips — request type chip (Bug fix / Enhancement / New app), target app name if any (plain name, never a repo), urgency, and "Submitted 3 days ago · #142" (a friendly short id, never the GitHub Issue number/link). A single big plain-language STATUS pill sits at the right of the header (e.g. amber "Action needed", green "Approved", Micron purple "Building") — this is the one status summary; its color follows the same status-type/attention tokens but is shown to the Submitter as a plain word, never the internal status_type name.
3. THE ATTENTION SLOT (conditional, only when state == Sent back): a full-width amber-edged callout card pinned directly under the header — the loudest element on the page. This is the respond-to-send-back surface (see states). When NOT sent back, this slot is absent (no empty box).
4. Plain-stage Status timeline: a vertical stepper of the six plain stages (Submitted · Spec drafted · Approved · Building · In review · Deployed) rendered as a Slack-thread-like sequence of immutable milestone rows — each with a status-type glyph (done check / current ring / upcoming dotted), a plain stage label, a plain one-line description, and a relative timestamp when reached. The current stage row is at full contrast; future stages are tertiary/ghosted. A 'Needs a human' state renders as a red-flagged overlay row injected at the current position (never its own happy-path step). Sent-back is shown on the timeline as the current stage having bounced back to 'Spec drafted', with the active work happening in the attention slot above.
5. 'What you told us' recap (collapsible, collapsed by default): the original Request fields + the AI interview Q&A in plain read-back form, so the Submitter can remember what they said. This is the Submitter-safe mirror of the Admin's grounded context — NO Draft spec internals, NO provenance tags, NO open-questions/assumptions block (those are Admin-only).
6. Footer: quiet help line ("Questions? Replies to the email notice reach the team.") — no destructive actions; Submitters cannot cancel (only Admins cancel, US38).

### Key components
- Submitter shell top bar (minimal: wordmark · My Requests back-link · user identity from Entra; no sidebar, no Cmd+K)
- Plain-language STATUS pill in the header — single source-of-truth status word, colored by the same attention/status-type tokens but labeled in Submitter vocabulary (Action needed / Submitted / Building / Approved / Deployed / Needs a human)
- Respond-to-send-back attention card (the conditional amber callout — a Submitter-facing relative of the Admin Attention badge; the one loud thing on the page)
- Admin note panel inside the attention card — the sent-back note shown verbatim, attributed plainly ('A reviewer asked:'), quoted/indented like a Slack message, never editable by the Submitter
- Plain-stage Status timeline = Slack-thread-style immutable milestone rows + Status-type glyph (done check / current progress ring / upcoming dotted circle); shape carries meaning so it reads color-blind
- Status header (edit-in-place) — only the current-stage row + the header status pill mutate between polls; every reached milestone below is immutable (holds the ADR-0004 line)
- Needs-a-human overlay row — red-flagged inline row at the current position, plain copy ('Paused — a person is taking a look'), never a column or happy-path stage

### States

- **Populated — in progress (happy path, no action needed)** — The common case. Status pill shows the current plain stage (e.g. Micron purple 'Building'). Attention slot is absent entirely. Timeline shows reached stages with done-checks + timestamps, the current stage with a progress ring at full contrast, future stages ghosted/dotted. Recap is collapsed. Calm, neutral+Micron purple register — zero loud color. This is the screen the Submitter sees most and it should feel reassuring and quiet.
- **Populated — Approved** — Status pill green 'Approved'. Timeline: Submitted ✓, Spec drafted ✓, Approved is now the current/most-recent done row with a check and 'Approved 2h ago — the team is setting up the work'. ### Accessibility

Status type is carried by glyph shape (dotted/ring/check/strike) first so it reads color-blind (1.4.1). The Needs-me count is an aria-live polite region so its changes are announced without stealing focus; the poll-freshness whisper is aria-live polite too, but the 3–5s diff-merge poll must NOT spam the live region (announce only meaningful count/state deltas, never every tick). G-nav and single-key actions never trap a screen-reader/keyboard user (they’re additive accelerators, every one has a focusable button equivalent). `prefers-reduced-motion` is honored for the side-panel slide, palette entrance, and card transitions — swapped for opacity cross-fades, not killed. Hit targets ≥24px even where icons render smaller (2.5.8). Hover-reveal tooltips also appear on focus (not hover-only) so keyboard users get the same shortcut hints.

## 8. Board (Kanban)

**Primary user:** admin

**Purpose.** The Control center's at-a-glance map of every Request and Work item in the Factory, grouped into the seven stage columns Intake → Spec → Architecture → Build → Review → Deploy → Done. Cards move themselves as stages advance — Admins don't drag between columns; they act on gates. In Stage 1 only the Intake and Spec columns hold live cards; Architecture/Build/Review/Deploy/Done render at reduced contrast with a quiet "Stage N — opens later" placeholder and no cards, filling in as Stages 2–6 ship. ## 9. List view (filters + saved views)

**Primary user:** admin

**Purpose.** The List view is the keyboard-fast, high-density second face of the one filterable Request/Work-item collection — the same data the Board/Kanban renders, regrouped from columns into a scannable vertical list. It exists for the Admin who wants to *triage by reading rows* rather than *survey by columns*: scan everything asked across the Factory on single lines (app · type chip · status-type glyph · attention badge · owner · stage · age), narrow it with flat AND filters (stage / app / type / owner / attention / label), and pin a handful of built-in Saved views (Waiting on me, Needs human, Active builds, Per app) as one-keystroke lenses. It is the Stage-1 workhorse surface where, day-to-day, an Admin lives between the Needs-me inbox and the Approval queue: the List lets them group-by stage/owner/app, sort by age or attention, multi-select rows, and fire single-key gate actions (Approve / Send-back / Cancel) without ever leaving the list — opening the Request/Work-item detail as an in-place right side-panel so the list stays as context. Board answers "where is everything?"; List answers "what, specifically, in priority order, and let me act on it."

### Layout
The inverted-L Control-center shell (reused verbatim, only the center swaps):

- LEFT SIDEBAR (persistent, chrome-dimmed, ~248px): Needs me (badge-counted) · Approval queue / Triage · Board / List toggle (the List item is the active/selected destination here, shared filter state with Board) · Saved views group with the 3–4 built-ins (Waiting on me, Needs human, Active builds, Per app — the last expands to a flat app sub-list) · Apps (registry + per-app feed channels) · Settings. ## 12. Per-app Progress feed (Slack-style) — C5

**Primary user:** admin

**Purpose.** Give an Admin a calm, scannable, per-app view of everything the Factory has done — grouped by app (Subject) the way Slack groups by channel — so they can watch builds happen and triage activity without opening GitHub. It is the second read-projection over the one two-axis `progress_event` log (ADR-0008): same data as the per-Request Progress timeline, regrouped by `subject_id`. North star: legible feed of immutable milestone summaries, loud only at gate boundaries and Needs-human, ambient (unread dots, never a count to clear). In Stage 1 the log writes from the start but reads thin — most apps show only Intake/Spec-era events plus the Spec-approval gate boundary — so the screen is built now and fills in as Stages 2–6 emit progress (LATER-surfacing per the foundation). This is a READ/TRIAGE feed, not a chat: no composer, no reactions, no emoji, no presence/typing — the immutable milestone cards stay read-only forever, holding the ADR-0004 line; action lives in the Approval queue / Request detail, never inline here.

## 13. Needs-me inbox (C6)

**Primary user:** admin

**Purpose.** The single clear-to-zero interrupt surface for an Admin — the only badge-counted, mark-readable, snoozable place in the whole product. It collects exactly the Tier-1 items that demand a human decision: the four human gates (Approve spec [live now], Sign ADRs, Approve merge, Approve deploy [skeleton/LATER]) plus the red Needs-human escalation, plus @mention handoffs from another Admin. It answers one question — "what is waiting on ME right now?" — and lets the Admin act (approve / send-back / cancel / open) or defer (snooze / mark-read) without leaving the list. In Stage 1 it is intentionally thin: the only gate that actually populates it is Approve spec; the other gate types and Needs-human are wired but rarely seen. 1. HEADER BLOCK (no card) — h1 "Notifications", one caption line of intent: "How the Factory reaches you. Quiet by default — Gate events and Needs-human always reach your inbox." Sets the restraint promise up front.

2. EVENT MATRIX — the heart. A 3-row × 3-column control grid titled h2 "What you hear about". Rows = the three coarse event classes; columns = the three delivery channels:
   • Row labels (left, with status-type glyph + one caption sub-line each):
     - "Gate events" — amber gate glyph — sub: "Approve spec · Sign ADRs · Approve merge · Approve deploy" — micro pill "TIER 1".
     - "Needs-human" — red blocked-overlay glyph — sub: "A build escalated and is waiting on a person" — micro pill "TIER 1".
     - "Progress feed" — Micron purple progress-ring glyph — sub: "Milestone summaries as apps move through stages" — micro pill "TIER 2".
   • Column headers (right, three toggles per row): "In-app" · "Email" · "Digest".
   • Cells = toggle switches. Tier-1 rows render In-app as an ON, locked/disabled switch with a tiny lock affordance + tooltip "Tier-1 events always appear in your Needs-me inbox" (you can route the *extra* email/digest copies, never silence the inbox itself). Email + Digest are mutually-aware on each row (choosing Digest dims Email to "rolled into digest"). Progress-feed row: In-app = the ambient unread dot (ON by default), Email = OFF/locked-low ("Progress is never emailed individually"), Digest = ON.
   A muted footnote under the matrix: "Email sends a single message per event; Digest batches still-unread items on your cadence below."

3. DIGEST CADENCE — h2 "Digest". A compact inline control: segmented control [ Off · Hourly · Daily · Off-hours only ] + a secondary "Send at" time picker (appears only for Daily) + a caption "Batches your still-unread Needs-me items — never re-sends what you've already cleared." A live preview line: "Next digest: tomorrow 9:00 AM · 3 unread items" (or "No unread items — nothing to send").

4. PER-APP FOLLOW LEVELS — h2 "Per app" + caption "Override the Progress-feed level for specific apps. Defaults to your matrix setting above." A dense Linear-style list (one app per scannable row): app avatar/initial · app name · subject channel hint (caption) · a right-aligned 3-segment follow-level control [ All · Gate + Needs-human · Muted ] mirroring the Slack per-channel follow model. A muted top row "All apps (default)" sets the baseline; per-app rows below show only apps that *differ* from default, plus an "+ Add app override" affordance that opens a Cmd+K-style app picker. Apps overridden to Muted show a small muted-bell glyph.

No explicit Save button by default — changes are optimistic and auto-persisted (per Linear/Slack settings convention) with a transient "Saved" micro-confirmation; a sticky footer Save/Discard bar appears ONLY if a write is in flight or fails.

### Key components
- Inverted-L shell (left rail + top header + calm centered ~760px column) — reused verbatim, board/list toggle and filter chips omitted
- Status-type glyph as the row identity — amber gate glyph (Gate events), red Needs-human overlay glyph, Micron purple progress-ring glyph (Progress feed) — shape carries the class so it reads color-blind
- Tier micro-pill (TIER 1 / TIER 2) — uppercase 11px micro label tying each row back to the consequence-tiering principle
- Toggle switch (semantic on/off) — compositor-only transform thumb slide; locked/disabled variant with lock glyph + tooltip for Tier-1 in-app cells
- Segmented control — for Digest cadence (Off/Hourly/Daily/Off-hours) and for the per-app 3-state follow-level (All / Gate + Needs-human / Muted), the literal Slack per-channel follow model
- Per-app follow-level list — Linear-style one-row-per-app dense list with avatar + name + right-aligned segmented control; shows only overrides + a default baseline row
- App-picker (Cmd+K-style) — '+ Add app override' opens the same command-palette local-search pattern scoped to the App registry
- Live digest preview line — 'Next digest: tomorrow 9:00 AM · N unread items' computed from current cadence + inbox state
- Optimistic auto-save + transient 'Saved' confirmation — no Save button in the happy path; sticky Save/Discard bar only on in-flight/error
- Hover-reveal shortcut tooltips + global :focus-visible Micron purple ring on every switch/segment
- Teaching empty state for per-app section — 'No per-app overrides — every app uses your default above'

### States
- **Loading (skeleton)** — Three-state discipline — structure not spinner. The matrix renders 3 ghost rows × 3 ghost toggle pills, the digest segment renders as a single shimmer bar, the per-app list shows 3 skeleton rows. Section headings and labels render immediately (they're static); only the values shimmer. No layout shift when values arrive.
- **Populated (default / restraint baseline)** — A brand-new Admin's defaults, encoding the restraint principle: Gate events → In-app ON+locked, Email ON, Digest off; Needs-human → In-app ON+locked, Email ON, Digest off; Progress feed → In-app (ambient dot) ON, Email locked-off, Digest ON. Digest = Daily 9:00 AM. Per-app section shows only the 'All apps (default): Gate + Needs-human' baseline row + the empty-state hint + 'Add app override'.
- **Customized** — An experienced Admin: Gate events email→off but digest→on; Progress feed digest→off (fully quiet ambient-only); Digest = Off-hours only. Per-app list shows 3 overrides — 'Atlas Billing: All', 'Internal Tools: Muted' (muted-bell glyph), 'Marketing Site: Gate + Needs-human'. Demonstrates the override-differs-from-default rendering.
- **Tier-1 silence attempt (constrained)** — Admin tries to toggle the locked In-app switch on a Tier-1 row; the switch does not move, a tooltip/inline caption surfaces: 'Tier-1 events always reach your Needs-me inbox — route the email and digest copies instead.' Reinforces guardrail without a hard error modal.
- **Saving (optimistic in-flight)** — A flipped toggle applies instantly (optimistic); a tiny inline spinner replaces the row's right-edge state for ~one poll cycle, then resolves to a transient 'Saved' micro-confirmation that fades. No global blocking state.
- **Save error (recoverable, specific)** — Persist fails: the affected control rolls back to its prior value (never a silent half-applied state), and a sticky footer bar appears naming the exact failed change — 'Couldn't save Progress-feed digest setting. Retry · Discard.' Specific + recoverable, never a generic toast.
- **Empty per-app overrides** — Teaching empty state inside the Per-app card: a muted line 'No per-app overrides yet — every app uses your default (Gate + Needs-human). Add one to mute a noisy app or follow a critical one closely.' with the '+ Add app override' control beneath.
- **Digest = Off** — Digest cadence set to Off: the live preview line reads 'Digest off — unread items stay in your Needs-me inbox only.' and all Digest-column toggles in the matrix dim to disabled with a caption 'Turn on a digest cadence to enable.' — keeping cause/effect legible.
- **Reduced-motion** — prefers-reduced-motion: toggle thumb and segment selection cross-fade opacity instead of sliding; 'Saved' confirmation fades without scale. No motion removed outright — feedback preserved.

### Interactions
- Click/tap a toggle → flips optimistically, transform-slide thumb (fast 120ms), auto-persists, transient 'Saved'; locked Tier-1 in-app toggles refuse and surface the guardrail caption
- Keyboard: every switch and segment is in the natural tab order; Space/Enter toggles a switch; Arrow Left/Right moves selection within a segmented control (Digest cadence, follow-level); focus shows the global 2px Micron purple :focus-visible ring with 2px offset, never obscured by the sticky header
- Choosing Digest on a matrix Email/Digest pair dims the sibling Email toggle to 'rolled into digest' with a caption — the two are mutually aware per row
- Digest cadence = Daily reveals the 'Send at' time picker inline (base 160ms reveal); the live preview line recomputes immediately from cadence + current unread count
- '+ Add app override' opens a Cmd+K-style local app picker (search over the loaded App registry); selecting an app inserts a new follow-level row defaulted to the matrix baseline, focus lands on its segmented control
- Setting a per-app follow level to Muted adds a muted-bell glyph and (if it matches default) the row can collapse back into the default; setting it away from default keeps it pinned in the override list
- Hover any control → reveal a shortcut/explanation tooltip (e.g. hover the lock → 'Tier-1 always reaches your inbox'); graduates mouse users toward understanding the tiering
- Cmd+K globally still works from this screen (jump to board/inbox/another app) — Settings is a normal Admin shell route, not a modal trap
- Sticky Save/Discard footer (error state only): Retry re-attempts the failed write; Discard rolls back to last-persisted; Enter = Retry, Esc = Discard
- G-nav still active (G I inbox, G B board, etc.) so an Admin can leave Settings by keyboard without reaching for the mouse

### Borrows from Linear / Slack
- Linear — Settings/notification preferences IA: a quiet centered reading-width form inside the same app shell (not a separate console), coarse toggles over granular rules, optimistic auto-save with no Save button in the happy path
- Linear — restraint as the default posture: a small number of consequence-tiered settings, not a Slack-style wall of per-event checkboxes; the screen embodies principle #10 (Restraint as a feature)
- Linear — keyboard-first parity: tab/Space/Arrow operation of every control, global :focus-visible ring, Cmd+K and G-nav remain live
- Linear — status-type glyph + tier semantics carried onto setting rows so event classes read by shape, color layered on top
- Slack — the per-channel follow-level model mapped exactly: channel = app (Subject), and the 3-state All / Gate + Needs-human / Muted control is the literal 'All / mentions / muted' Slack notification preference per channel
- Slack — the three-channel delivery framing (in-app / email / digest) echoing Slack's 'where do notifications go' settings, but trimmed to three coarse rows
- Control-center brief — the three coarse settings + tier-by-consequence model (Tier-1 interrupts, Tier-2 ambient, Tier-3 nothing) and digest-gated-on-unread email with zero push infra
- Clear-to-zero inbox concept — Settings is explicitly the dial for the Needs-me inbox (Tier-1) vs the ambient feed (Tier-2/3), so it borrows the inbox-vs-feed distinction as its organizing axis
