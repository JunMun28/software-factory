# Design System — consolidated tokens & component inventory

> The detailed token tables live in the design foundation ([README.md](README.md)). This is the consolidated, build-ready summary.

> **Brand: Micron.** Accent = **Micron purple `#BD03F7`** (focus ring `#CE57F9`; `accent-strong #9B02C9` for small text on light). Black/white neutral base. Purple gradients only at brand moments (login hero, empty states, logo lockup); working surfaces flat. Functional status colors kept and are not brand: amber `#E3B341` (gate needs you), red `#F2545B` (Needs-human), green `#3FB950` (done). See README → "Brand: Micron alignment".

## Principles
- Quiet by default, loud only when it matters. The whole UI lives in a calm neutral+Micron purple register. Only two things are ever allowed to be loud: a gate waiting on you (amber) and Needs-human (red). Tier by consequence everywhere — Tier-1 (gates, Needs-human) interrupts via the Needs-me inbox; Tier-2 (feed activity) is an ambient unread dot; Tier-3 (a card merely changing column) is nothing. At most one amber and one red appear per screen.
- Structure felt, not seen. Hairline low-contrast borders, whitespace, and the 8px rhythm carry hierarchy — not boxes and rules. Chrome (sidebar, rails, headers) recedes at text-secondary contrast; the work (a board, a Draft spec under review, a Progress thread) sits at full contrast. No boxes-in-boxes, no heavy drop-shadows.
- Keyboard-first, mouse-optional. Cmd+K, G-nav, and single-key gate actions are the primary input model for Admins. Hover-to-reveal-shortcut tooltips graduate users from mouse to keyboard. Every primary Admin action has keyboard parity with its button. (Submitters get one power-keystroke — Cmd/Ctrl+Enter to submit — and are otherwise mouse/touch-first.)
- Density with grouping, not noise. One scannable line carries app · type chip · status-type glyph · attention badge · owner · stage · age. Group related info; separate zones with whitespace; optimize for the daily-repeat expert, not the novice.
- Felt speed without a sync engine. Optimistic UI on gate actions + signal-based granular re-render + diff-merge polling (keyset ?after=<event_id> cursor) ≈ 80% of Linear's feel on a polling FastAPI+Angular stack. Never a whole-board/feed refresh-flash; only changed cards/rows transition.
- Grounded approval beats fast approval. Provenance tags (from: Q2) / (ASSUMPTION — not stated) and a pinned Open-questions/Assumptions block are the anti-rubber-stamp mechanism — kept unavoidable in the reviewer's eye-path, directly between the title and the Approve button. Friction is good at exactly one place: the irreversible Approve, whose confirm names the GitHub side effects.
- Status type by shape, status by color. A glyph (dotted/ring/check/strike) carries the type so it reads color-blind; color is a signal layered on top, never the only cue (WCAG 1.4.1). Needs-human is a red overlay badge, never a column or a type. All filters/views query the status_type, never the seven column strings.
- Two faces, one system. Submitter (guided, light-default, 16px body-lg, mobile-first, plain-stage vocabulary) and Admin (dense, light-default, 14px, keyboard) are forked shells under one auth + one token set. Light is the product's default theme for both faces; dark is an opt-in. Submitters never see GitHub, repo names, PR/Issue numbers, the Draft-spec internals, provenance tags, or Control-center words.
- Hold the ADR-0004 line. The Slack metaphor must not smuggle back streaming: no presence, no typing, no per-milestone push, no chat.update on historical summaries. Milestone summaries are immutable; only the per-thread/Work-item status header edits in place. (The one legitimate streaming exception is the Submitter Intake interview — direct in-app LLM calls with the user waiting — a different surface from the Control-center feed.)
- Restraint as a feature. Skip WIP limits, reviewer rotation/honeypots, push infra, OR-filters, user-authored views, predictive widgets, IndexedDB sync. Build the keyboard-fast core for a handful of Admins; defer the rest. The App registry and Settings are deliberately the quietest screens in the product.
- Clear-to-zero vs ambient is the organizing axis. The Needs-me inbox is the only badge-counted, mark-readable, snoozable surface meant to reach zero (Tier-1 only). The per-app Progress feed is never 'done' — binary ambient unread dots, never a count to clear. This Linear-inbox-vs-Slack-feed distinction is the synthesis the whole product is built on.

## Color — light

The default theme for BOTH the Submitter and the Admin/Control-center surfaces (the product's primary theme). Neutrals carry a faint cool/Micron purple tint, never pure gray/white. bg-canvas #FBFBFC (warm near-white) · bg-surface #FFFFFF (cards) · bg-input #F4F4F6 (wells, hover) · border-subtle #E8E9ED (hairline dividers, card edges) · border-strong #D0D2DA (input/focused edges) · text-primary #16181D (headings/copy, never pure black) · text-secondary #5A606B (labels, metadata, timestamps) · text-tertiary #9DA3AE (disabled, faint hints) · accent #BD03F7 (Micron purple, primary action) · focus-ring Micron purple at >3:1 on all surfaces. Status/attention (always paired with glyph+label): status-triage #6B7079 · status-started #BD03F7 · status-completed #1A7F37 (green — the ONLY green, used once on success) · status-canceled #9DA3AE · attention-gate #9A6700 (darker light-theme amber so the label clears 4.5:1 on white — NOT the dark #E3B341) · attention-blocked #D1242F (red, clears 4.5:1 on white). Amber and red are the only loud colors and never alone — they carry a glyph shape and a plain word.
Verify all pairs to 4.5:1 text / 3:1 non-text before locking. Token-swap theming: :root = light, [data-theme="dark"] = dark; a non-compliant pair can't be wired because components reference only semantic tokens.

## Color — dark (opt-in)

Optional dark mode — an opt-in theme (no longer the Admin default; applied via `[data-theme="dark"]`). Near-black with a faint cool/Micron purple tint, never pure black/white. bg-canvas #0B0C0E (app background) · bg-surface #141518 (cards, columns, panels) · bg-surface-raised #1B1D21 (popovers, menus, modals, Cmd+K) · bg-input #202329 (inputs, hover wells, focused-row wash) · border-subtle #23252A (hairline dividers, card edges) · border-strong #33363D (input borders, focused edge) · text-primary #F2F3F5 (off-white, never #FFF — avoids glare) · text-secondary #9DA3AE (labels, metadata, timestamps, recessed chrome) · text-tertiary #6B7079 (disabled, faint hints, inert placeholder columns) · accent #BD03F7 (Micron purple, brand/primary action) · focus-ring #CE57F9 (focus-visible outline, >3:1 on all surfaces). Status/attention (paired with glyph+label): status-triage #8A8F98 · status-started #BD03F7 · status-completed #3FB950 · status-canceled #6B7079 · attention-gate #E3B341 (amber) · attention-blocked #F2545B (red). Amber and red are the only loud colors in the whole product, appearing at most once each per screen; the diff-view green/red gutter is the single exception where green/red mean add/remove (paired with +/- glyphs), disambiguated from status color by context.
Three conceptual knobs (base / accent / contrast) make a high-contrast theme cheap to ship.

## Typography

Font: "Inter var", "Inter", "Segoe UI", system-ui, sans-serif (Segoe first = native to the Windows shop). Mono: "Cascadia Code", "JetBrains Mono", ui-monospace, Consolas, monospace — for PR/Issue refs, branches, commits, repo owner/repo strings, diffs, and SPEC.md headings only. Weights kept tight: 400 body, 500 UI labels/buttons/emphasis, 600 headings (nothing heavier). Negative tracking on large sizes is the Linear tell.
Scale (token · size/line-height · tracking · use):
- display · 32/1.25 · -0.02em · Intake hero
- h1 · 24/1.33 · -0.015em · detail title, Request title, Submitter page titles
- h2 · 20/1.4 · -0.01em · section headings
- h3 · 16/1.5 · -0.005em · card titles, panel headings, row titles
- body · 14/1.6 · 0 · ADMIN DEFAULT — board, list, feed, forms
- body-lg · 16/1.6 · 0 · SUBMITTER DEFAULT — form copy, Draft-spec reading, chat
- caption · 13/1.5 · 0 · metadata, timestamps, feed context line, receipt stub
- micro · 11/1.4 · +0.01em · badge/pill/tier labels (uppercase)
Plain-stage vocabulary is law on the Submitter face: only {Submitted, Spec drafted, Approved, Building, In review, Deployed, Needs your input/Needs a human, Cancelled}. Admin vocabulary (status_type, Work item, gate, Triage, send-back, SPEC.md, PR) never leaks Submitter-side.

## Spacing

4px base with an 8px-rhythm bias: 0 · 4 · 8 · 12 · 16 · 20 · 24 · 32 · 40 · 48 · 64 · 80. Card padding 16/24 (Submitter forms 24/32); element gap 8/12; column gutter 16; Kanban columns ~300px fixed width; section gaps 48+ (settings/Submitter reading views). Inbox/triage rows 40–48px tall; list/feed rows 40–44px. Sidebar ~232–248px; top header ~52px; in-place right side-panel ~440–520px (expandable to a centered ~960px reading width). Centered reading views: Submitter detail/confirmation ~560–720px, My Requests ~760px, Approval-queue spec pane ~880px, Settings ~760px; Admin centered detail caps ~1200px. The board and per-app feed are full-bleed horizontal (board) / channel-body reading-width-capped ~720px (feed); never a stretched mobile view. Field structure repeats like a ruler: label → helper caption → input, vertically on the 8px rhythm.

## Radii & elevation

Radii (small and crisp, never bubbly): xs 2px (pills/badges) · sm 6px (inputs, buttons, menu items, nav rows) · md 8px (popovers, small cards) · lg 12px (cards, modals, columns, auth card) · full 9999px (avatars, status dots, unread dots).
Elevation — depth cue differs by theme. Dark: surface-lightness step + a 1px INSET hairline border is the real depth cue (shadows read weakly on near-black). Light: true soft shadows. elev-0 flat + border-subtle; elev-1 inset border + 0 2px 4px rgba(0,0,0,.4) dark / 0 1px 3px rgba(16,24,40,.10) light (cards, raised rows); elev-2 0 4px 24px rgba(0,0,0,.6) + inset dark / 0 8px 24px rgba(16,24,40,.12) light (Cmd+K palette, right side-panel, modals). Selected/active states use a bg-input wash + a 2px accent left-marker (nav, selected cards) rather than a shadow. On the smallest Submitter breakpoints, cards may shed their border/shadow and blend into the canvas where the edge stops earning its keep.

## Motion

Animate ONLY transform + opacity (compositor-only). Asymmetric: summon instant, dismiss fade. Durations: instant 80ms (hover/press) · fast 120ms (DEFAULT — menus, tooltips, state/badge changes, card hover, pill cross-fade) · base 160ms (popover/drawer/side-panel open, optimistic card move, regroup) · slow 220ms (modal/palette/route transition). Stay in the 100–150ms band for most feedback. Easing: ease-out cubic-bezier(0.16,1,0.3,1) for entrances (default); ease-in-out cubic-bezier(0.65,0,0.35,1) for state-to-state and cards sliding between columns; ease-in cubic-bezier(0.4,0,1,1) for exits. Motion is feedback, not decoration.
Hard rules: (1) The 3–5s diff-merge poll must NEVER auto-animate on every refresh — only genuinely changed cards/rows transition; existing content never re-animates; never a whole-surface flash. (2) Mandatory prefers-reduced-motion guard — swap slides/scales for opacity cross-fades, never just kill the transition (typing-dot pulse → static "Working…"; token streaming → whole-question reveal; hero check scale → fade; segment slide → cross-fade). (3) Optimistic actions acknowledge within instant(80ms) so a click feels registered before the server responds.

## Iconography

De-iconified, monochrome line icons inheriting currentColor, scaled down, no colored backgrounds, no colored fills. The load-bearing set is the STATUS-TYPE GLYPHS (shape carries type so it reads color-blind): dotted circle = Triage (Intake/early) · progress ring whose fill = position in the 6-stage run = Started (Spec→Deploy in progress) · solid check = Completed (Done/gate passed/Deployed) · strike = Canceled. Needs-human = a small red flag OVERLAY badge on the glyph (never its own glyph, never a column). Submitter face reuses the exact same glyph set, relabeled to plain stages. Other icons: a monochrome assistant glyph for AI turns (NOT a human-face avatar), a Factory line-mark for the brand lockup, a de-iconified channel glyph + binary ambient unread dot for per-app feed channels, the official 4-color Microsoft logo as the ONE sanctioned color exception (a recognized trust signal, on the SSO button only). All icons get a ≥24px hit area even when visually smaller (WCAG 2.5.8).

## Accessibility

WCAG 2.2 AA enforced at the token level (4.5:1 text / 3:1 non-text), verified on every pair before locking; recessed text-secondary chrome checked against its surface; amber/red verified as non-text 3:1 AND never carrying meaning by color alone.
Color never alone (1.4.1): status_type is carried by glyph SHAPE first (dotted/ring/check/strike), color second; attention badges and Needs-human always pair their hue with a plain word ('Approve spec', 'Needs human'); the diff green/red gutter is paired with +/- glyphs.
Focus: global :focus-visible { outline:2px solid var(--focus-ring); outline-offset:2px } on every interactive element and the active row/card; focus never obscured by sticky headers, the action bar, the side-panel, or the palette (2.4.11) — scroll-padding reserves their height; opening/closing the side-panel or palette traps then restores focus to the originating control/row.
Keyboard: full operability is the primary Admin model, not an afterthought — skip-to-content first, logical tab order (header → sidebar → center → cards), every action reachable by Tab+Enter with the single-key grammar as an additive accelerator; no mouse-only or keyboard-only-discoverable paths; Esc closes panels and returns focus.
Semantics: command palette = role=dialog/aria-modal with listbox navigation + aria-selected announcing each result and its shortcut; queues/lists as semantic list/grid with aria-selected and group headers as labelled regions with aria-expanded; the plain-stage timeline as an <ol> with aria-current="step" and per-node text state ('Building — not started yet'); switches as role=switch with locked state via aria-disabled + aria-describedby explaining why.
Live regions (polite, never spammed by the 3–5s poll): announce only meaningful deltas — a new gate ('Now 2 items waiting on you'), an optimistic state change/rollback (naming the specific failed setting/step), a newly reached stage; broadcast (gate/Needs-human) may announce assertively. The Intake transcript announces each completed AI question once it finishes streaming (NOT token-by-token); the typing indicator carries an accessible 'Assistant is writing' label.
Targets ≥24px hit area even where glyphs/dots/checkboxes render smaller (2.5.8); ≥44px on the Submitter/mobile touch face. Reduced-motion honored as opacity cross-fades, never just disabled. Tooltips fire on focus and are dismissible (1.4.13). Errors/empty/snoozed states expose readable text (not icon-only) so AT conveys 'cleared' / 'snoozed until X' / the specific recoverable error.
Submitter face: plain-language everywhere (no GitHub/Control-center jargon) is itself a cognitive-accessibility feature; 16px body-lg; ownership-scoped so a Submitter can never even transiently see another user's rows; the Microsoft logo is aria-hidden (decorative, redundant to the text label). A high-contrast theme ships cheaply via the three-token model.

## Angular implementation approach

FastAPI backend + Angular frontend (ADR-0007), polling now / SSE later. One Angular app, two forked shells (Submitter / Admin) selected from the role resolved off the validated Microsoft Entra session at login — a single auth + single token set, routed at the root so a Submitter hitting a Control-center route is bounced server-side (authorization is never a client toggle).
Token architecture: three tiers primitive → semantic (intent-named, theme-aware: bg-canvas, text-primary, border-subtle, accent, focus-ring, status-*, attention-*) → component. Plain CSS custom properties; :root = light, [data-theme="dark"] = dark, default from prefers-color-scheme. Components reference ONLY semantic tokens, so a non-compliant pair physically can't be wired and theming/high-contrast is a variable swap (the three-knob base/accent/contrast model).
Felt-speed engine (no sync engine): (1) Optimistic UI — gate actions (Approve/Send-back/Cancel) apply to local signal state and render within instant(80ms), then reconcile on the next poll, rolling back only on failure with a specific recoverable error. (2) Signal-based granular re-render — Angular signals + OnPush + trackBy/computed so only the changed card/row/count re-renders; never a list-wide redraw. (3) Diff-merge polling — a 3–5s poll on the keyset ?after=<event_id> cursor (the SAME cursor the feed_read_state watermark uses) merges only new/changed progress_events by id; existing immutable cards never re-animate; that cursor is the exact seam SSE drops into later without reopening ADR-0007.
Data model is one read source: the Control center reads the single append-only two-axis progress_event log (ADR-0008) — per-Request timeline = filter by request_id, per-app feed = filter by subject_id (fan-out-on-read, one writer). The UI must encode the ADR-0004 guardrail: only the status header is edit-in-place; historical milestone cards are immutable components with no edit affordance.
Motion in Angular: animate only transform/opacity via the Web Animations API / CSS, gated behind a global prefers-reduced-motion guard that swaps slides for opacity cross-fades. The poll must be wired so it never triggers entrance animations on unchanged nodes (key on event id, animate only on insert).
Keyboard layer: a global Cmd+K command-palette overlay (role=dialog, focus-trapped, listbox with aria-activedescendant) over the already-loaded local store; a G-nav prefix directive; single-key action directives on the focused item; hover-AND-focus shortcut tooltips. Every accelerator has a focusable button equivalent (additive, never the only path).
Components: Standalone Angular components, OnPush change detection, signal stores for the local board/list/inbox/feed collections so optimistic mutations and diff-merge are cheap. Right side-panel is an in-place overlay routed via outlet/state (not a full route change) so the center stays mounted as context.
