# Claude-Design Mockup Brief

> Handoff for building the actual UI mockups. Pair this with [design-system.md](design-system.md) (tokens) and [screens.md](screens.md) (full specs). Build the mockups, not the functional app.

## Design system in one breath
- Quiet by default, loud only when it matters. The whole UI lives in a calm neutral+Micron purple register. Only two things are ever allowed to be loud: a gate waiting on you (amber) and Needs-human (red). Tier by consequence everywhere — Tier-1 (gates, Needs-human) interrupts via the Needs-me inbox; Tier-2 (feed activity) is an ambient unread dot; Tier-3 (a card merely changing column) is nothing. At most one amber and one red appear per screen.
- Structure felt, not seen. Hairline low-contrast borders, whitespace, and the 8px rhythm carry hierarchy — not boxes and rules. Chrome (sidebar, rails, headers) recedes at text-secondary contrast; the work (a board, a Draft spec under review, a Progress thread) sits at full contrast. No boxes-in-boxes, no heavy drop-shadows.
- Keyboard-first, mouse-optional. Cmd+K, G-nav, and single-key gate actions are the primary input model for Admins. Hover-to-reveal-shortcut tooltips graduate users from mouse to keyboard. Every primary Admin action has keyboard parity with its button. (Submitters get one power-keystroke — Cmd/Ctrl+Enter to submit — and are otherwise mouse/touch-first.)
- Density with grouping, not noise. One scannable line carries app · type chip · status-type glyph · attention badge · owner · stage · age. Group related info; separate zones with whitespace; optimize for the daily-repeat expert, not the novice.
- Felt speed without a sync engine. Optimistic UI on gate actions + signal-based granular re-render + diff-merge polling (keyset ?after=<event_id> cursor) ≈ 80% of Linear's feel on a polling FastAPI+Angular stack. Never a whole-board/feed refresh-flash; only changed cards/rows transition.
- Grounded approval beats fast approval. Provenance tags (from: Q2) / (ASSUMPTION — not stated) and a pinned Open-questions/Assumptions block are the anti-rubber-stamp mechanism — kept unavoidable in the reviewer's eye-path, directly between the title and the Approve button. Friction is good at exactly one place: the irreversible Approve, whose confirm names the GitHub side effects.

## Non-negotiable guardrails
- ADR-0004 (milestone summaries, not streaming) is well-held on the Admin side: C4 Detail and C5 Feed both explicitly render milestone cards as immutable/historical, restrict edit-in-place to the status header only, and ban presence/typing/reactions/composer. Strong adherence — keep the explicit 'status header is the ONLY edit-in-place surface' callout in every timeline spec.
- The S2 streaming exception is correctly scoped and justified (Submitter-side, synchronous in-app LLM per PRD line 75) and is NOT smuggled into any Control-center surface. The only risk is documentation drift — see the inconsistency about centralizing the streaming boundary into one statement.
- Notification restraint (Linear over Slack volume) is well-respected: C6 Needs-me is the single clear-to-zero badge-counted surface (Tier-1 only); C5 feed uses a binary ambient unread dot, never a count; C8 caps settings to 3 coarse classes × 3 channels + per-app follow level, with Tier-1 in-app locked-on. This matches the brief's 'two surfaces, three coarse settings, zero push infra' exactly. Do not let any spec add per-event toggles or a second badge-counted feed.
- Polling-now (ADR-0007) is consistently honored: every Admin surface specifies the 3–5s diff-merge poll with 'never a whole-board/feed flash', optimistic-then-reconcile, and an 'Updated Ns ago' whisper; the Submitter side uses a gentler cadence. No spec assumes SSE/websockets. Keep the keyset '?after=<cursor>' framing so the SSE seam stays drop-in.
- 'Handful of admins — do not over-build' is respected: AND-only flat filters, 3–4 built-in saved views (no user-authored views), no WIP limits, no OR-filters, no multi-team, mobile explicitly de-prioritized for Admin surfaces. The control-center brief's 'Rejected by critic' (a separate cross-app Activity inbox) is correctly absent. Hold this line against any future request for custom views or a dense activity console.
- Color-as-signal discipline (amber=gate, red=Needs-human, one loud thing per surface) is stated in every spec and reinforced by status-by-SHAPE glyphs for color-blind legibility — strong and consistent. The one place to police: the diff view in C4 is the SANCTIONED exception where green/red mean add/remove (paired with +/- glyphs); ensure no other surface reuses green/red for add/remove semantics, and that S3's single green hero check is the only green in the Submitter app.
- Pre-approval action bar is EXACTLY three buttons: **Approve (A) · Send back (S) · Cancel (C)** — never a fourth, never relabeled. 'Send back' returns the Request to the *Submitter* for more info (RequestLifecycle PendingApproval→SentBack; no repo exists yet). 'Needs human' is NOT a button here — it is an automatic red escalation *status/overlay* the Factory raises only on a running Stage 2–6 build that times out or fails its gate; once a card is in that state its action bar swaps to the Recovery cluster (Retry / Retry-with-note / Send-back-to-stage / Take-over / Cancel). Do not place a 'Needs human' action on any spec-approval surface (C3/C4/C6). Note: an image-gen pass mistakenly rendered the 'Cancel (C)' button as 'Needs human (C)' — keep the label literally 'Cancel'.
- Persist-first / resilience (PRD hardening #4, ADR-0006) is correctly surfaced as UX: S2/S3 state the Request is safe even mid-spec-generation, S3 documents the GitHub-Issue-pending step as INVISIBLE to the Submitter, and C3/C4 show the per-step Approve ledger (repo_ready → spec_pr_open → stage2_fired) with idempotent Retry-from-failed-step. This faithfully translates the resumability ADR into recovery UX — keep it.

## Per-screen mockup notes

### 1. S0 — Login / SSO landing (Microsoft SSO front door)  (submitter)
Nail these in the mockup: (1) RESTRAINT IS THE DESIGN — resist adding a hero, illustration, feature list, or second button. One card, one Micron purple 'Sign in with Microsoft' button, one reassurance line. The empty canvas is the point; do not fill it. (2) This is the SHARED pre-fork screen, so render it in the LIGHT theme by default (Submitter-facing, body-lg/16px, approachable), but build it as a pure token swap so the dark variant exists for Admins/prefers-color-scheme:dark — show the light version as the primary mock. (3) The Microsoft 4-color logo is the ONLY saturated color allowed on the screen — it's a deliberate, sanctioned exception to the monochrome-icon rule because it's a recognized trust signal. Everything else stays neutral+Micron purple. Do NOT introduce amber or red anywhere — even errors here are calm/recoverable-toned, since amber=gate and red=Needs-human are reserved for deep in the product. (4) Make the SSO button the single state-bearing surface: show at least the idle state plus a small inset of the loading state ('Opening Microsoft…' with a spinner) and one specific recoverable error banner ('Sign-in was cancelled. Try again when you're ready.' or popup-blocked) to demonstrate the three-state discipline. Errors must read as helpful, never alarming. (5) Typography sells the brand: Inter, wordmark in 600 with visible negative tracking, microcopy in soft gray — this first frame must telegraph 'same product' as the dense Admin screens. (6) Keyboard-first credibility even here: show the Micron purple :focus-visible ring on the Sign-in button. (7) Copy is plain-language for a non-technical Submitter — zero internal vocabulary (no 'Control center', 'Draft spec', 'Work item', or GitHub). 'no new password' and 'Contact IT' are the load-bearing reassurance lines; keep them. (8) Mobile-first: also show the <640px frame where the card sheds its border and blends into the canvas with a full-width 44px button — prove it's reflow-proof.

### 3. Intake interview (AI chat) — S2  (submitter)
Nail these in the mockup: (1) This is the SUBMITTER face — render it in LIGHT theme, 16px body-lg, plain language, no sidebar, no Cmd+K, and ZERO GitHub/Control-center/'spec' vocabulary anywhere. (4) One question at a time, warm and short; reuse the Slack thread layout (left assistant / right user, timestamps) but strip everything chatbot-y: no model picker, no history rail, no markdown toolbar, no reactions, no emoji, no human-face avatar for the AI. (10) Mobile-first mindset: the desktop centered column should clearly collapse to a full-bleed phone conversation with a docked composer and full-width CTA.

### 4. S3 — Submission confirmation  (submitter)
Nail the RESTRAINT — this screen earns its keep by doing almost nothing. Resist adding a Draft-spec preview (Admin-only, ADR-0003), a progress firehose (ADR-0004), or any Control-center words ('Subject', 'Work item', 'gate', 'SPEC.md'); Submitters never see those. ONE spot of green only: the hero check. Everything else stays in the calm neutral+Micron purple register; the single primary CTA is the only saturated Micron purple fill. Use the SHARED status-type glyph set (solid-check here) but label it in PLAIN-stage vocabulary so it visually rhymes with the Admin board without leaking it — this continuity is the 'two faces, one system' payoff. The mini-tracker must show all six plain stages with only 'Submitted' lit, and the labels must exactly match US14's words (Submitted, Spec drafted, Approved, Building, In review, Deployed) so S4/S5 feel familiar; do NOT show 'Needs a human' here (it's a branch, not a default). Echo a concrete, copyable Request reference (REQ-2041 in mono) — the 'receipt stub' is what makes 'received' feel real. Honesty over hype: the what-happens-next sentence must name the approve/send-back fork and the email promise plainly (US15), and must NOT promise GitHub visibility. Persist-first means 'received' is true even mid-spec-generation — design the loading state as a skeleton on the receipt row + tiny 'Saving…' spinner, never a blocking gate over the confirmation. Make 'Track this request' the focused, obvious next move (focus on mount, Enter advances); 'File another' is deliberately quieter. Build it light-theme, body-lg 16px, mobile-first: on phones the tracker goes vertical and buttons stack full-width with the primary on top. Accessibility: aria-live announces the received reference, the check is shape+word not color-alone, the tracker is a labelled group with per-node text state, and prefers-reduced-motion cross-fades the hero instead of scaling it. The feeling to land: a quiet exhale — 'done, it's safe, here's where to watch' — not a fireworks 'success!' moment.

### 5. My Requests (Submitter list) — S4  (submitter)

(4) Honor the foundation's restraint: no table headers (row-cards, not a grid), one AND-only Active/All toggle, no saved views, no density toggle. ### 12. Per-app Progress feed (Slack-style) — C5  (admin)
