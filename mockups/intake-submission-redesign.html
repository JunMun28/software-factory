Submission journey redesign — visual plan

Approach 2: refined four-step wizard plus a live "what we understood" panel. Light warm canvas, one purple accent, plain language. Annotations above each screen say what changes and which audit findings it fixes.

Type shown in Inter as a stand-in for Archivo. Technical refs use JetBrains Mono. All body text ≥16px, controls ≥44px tall, everything keyboard-reachable.

Step 1 · /submit/new

Describe — a slim, calm start

Three inputs, not six. Reach, impact and urgency are gone from the form — the interview asks them only when they matter. The draft autosaves, so a refresh loses nothing.

  • type-first, then only relevant fields
  • keyboard-first app picker
  • draft autosave
  • errors are visible, never silent

Step 2 · /submit/:id/interview

Questions — one at a time, with the understanding panel

One way to answer each question: options or text, never both fighting. Progress is honest. Skipping tells you what we'll assume. On the right, "What we understood" grows with every answer — this same panel becomes the spec on the next step.

  • single input surface per question
  • honest progress count
  • skip shows its consequence
  • no re-asking what the form answered
  • live understanding panel

Step 3 · /submit/:id/review

Check — the real draft spec, before anything is sent

The understanding panel matured into the draft spec. Every line says where it came from; every assumption is flagged and asks for a decision. This is the "did we get this right?" moment the brief is built around — it needs the backend to draft the spec when the interview ends, not after submit.

  • real spec lines with provenance
  • assumptions ask to be confirmed
  • attachments visible
  • failures show a recovery path
