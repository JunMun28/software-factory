architecture overview
Autonomous ◆ Governed ◆ Full SDLC

An AI pipeline that ships code —
with humans on the irreversible boundaries.

Software Factory carries one unit of work through the whole software lifecycle as a chain of narrow, single-responsibility agents — requirements → architecture → test-first build → review → deploy. Each agent hands a validated artifact to the next, machine-checked gates catch cheating, and a human approves every step that can't be undone.

02

How a request flows

From a non-technical request to merged code — the whole journey, and who is in control at each turn.

1

A Request comes in the front door Submitter

Someone files a bug fix, enhancement, or new-app request in the Intake form. A short real-time Intake interview asks ~3–4 follow-ups to fill the gaps — no GitHub needed.

2

The brain drafts a spec Brain seam

The Stage-1 brain turns the enriched request into a Draft spec, stored in the database — not yet a repo file.

3

An Admin approves the spec Admin gate

In the Control center, an Admin reads the draft and approves. That creates / picks the Subject repo, commits SPEC.md to a fresh branch — the Work item is born — and starts Stage 2.

4

Agents build it test-first Runner seam

Architecture → failing tests → implementation → review run as commits on the branch. Every agent call is bounded; each stage posts a milestone summary to the append-only progress log.

5

Gates catch cheating automatically Automated

The RED gate proves tests fail for the right reason; GREEN + test-isolation proves the suite passes and the implementer never touched the tests. Reward-hacking can't slip through.

6

A human merges, then ships Admin gate

The Admin approves the Merge gate (PR → main) and the Deploy gate (main → production) as themselves — real GitHub approvals on protected branches. The Factory never bypasses GitHub.

Two seams: offline by default, real on demand

The whole factory runs offline with deterministic stand-ins. Flip an env var to swap in a real LLM — the domain model never changes. Deterministic seams are disposable; the domain model is not.

Brain seam

Stage 1 — the intake interview & draft spec.
default
ScriptedBrain — deterministic, no LLM calls.
real
ClaudeBrain — claude_brain.py, falls back to scripted if the model call fails.
flip with FACTORY_BRAIN=claude

Runner seam

Stages 2–5 — architecture through review.
default
simulator.py — a tick-driven simulation of the CI agents.
real
ClaudeRunner — runs in a per-request git workspace with live gates.
flip with FACTORY_RUNNER=claude · pick CLI with FACTORY_CLI= codex (default) | claude
06

The gates

Rules are guaranteed at the git / orchestration layer — checked against the artifact, not the agent's good behavior. The test-isolation gate is the load-bearing one.

✓

RED gate automated

New tests collect cleanly, fail as assertions (not runtime errors), and leave the existing suite green. "Fail for the right reason."

✓

GREEN + test-isolation automated

The full suite passes and the frozen tests/ hash is untouched — an implementer that weakens its tests to "pass" is caught and escalated.

✓

Review gate automated

A review summary artifact must exist in the work item's workspace — a structural check the pipeline enforces.

◆

Merge gate human

An Admin approves the PR into main as themselves. A protected-branch approval — GitHub blocks self-approval and still requires all CI checks.

◆

Deploy gate human

Ship to production by approving a main → production promotion PR. Same protected-branch mechanism as merge.

◆

Spec approval & Sign ADRs human

The two early human gates: an Admin approves the draft spec (S1) and signs off the architecture (S2) before a single line is built.

▲
No silent stranding.

Any gate failure, timeout, or crashed stage stops the pipeline and flags the work item needs_human — there is no automatic retry. An Admin then picks a recovery action: Retry, Retry with a note, Send back, Take over, or Cancel. Every stage is re-runnable against the current PR state (resumability).

07

Start the app

Orchestration is Task (cross-platform). Activate the pinned Node once per shell (.nvmrc → 24.15.0, nvm use), then run a single recipe.

software-factory — task dev
# everything, offline (simulator ticks every 8s)
task dev
  → API        http://localhost:8000
  → Intake     http://localhost:4201
  → Console    http://localhost:4202

# open the front door, sign in as a Submitter,
# or "as a reviewer" for the Control center.

# verify before merging anything
task verify     ✓ VERIFY PASSED
# lint + pytest + vitest + ng build + smoke

# re-seed the demo world / reset the db
task reset

# production-shaped stack (nginx + volume)
task up        → http://localhost:8080

# ── run the REAL factory ──────────────────
cd api && FACTORY_BRAIN=claude FACTORY_RUNNER=claude \
  uv run uvicorn app.main:app --port 8000

Single uvicorn worker only — the tick loop and pipeline threads assume one process and SQLite has one writer. Don't scale replicas without an ADR.
